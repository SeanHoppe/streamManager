## TASK A — Soak cassette + replay tiers (P0)

```
Cut quota cost of routine soaks and remove upstream rate-limit variance
from latency numbers. v1.1 30-min ship-gate soak burned 60 real
`claude -p` calls; CI cannot run it; rate-limit jitter corrupts p95.
This task introduces a three-tier soak model so CI can run end-to-end
plumbing tests for free, weekly cassette refresh catches model-side
envelope drift on a cheap model, and the ship-gate soak remains the
sole source of truth for ADR-5 absolute latency.

Reference:
- tools/soak_driver.py — current single-mode driver
- src/stream_manager/cli_pool.py — CliPool / CliWorker (DO NOT touch
  these symbols; replay path must stub at a higher layer)
- src/stream_manager/cli_governance.py — escalation entry point
- reports/soak-20260503T101758Z.md — v1.1 ship-gate baseline format

DO NOT touch (v1.1 protected surface):
- CliPool / CliWorker / .bridge/cli-pool.pids — Task J
- _install_lazy_hydrator / GovernanceEngine.hydrated / tools/perf_probe.py
  — Task I
The replay path must inject a fake CLI worker at the CliPool boundary
(or above), not by editing CliPool internals.

Three tiers:

1. REPLAY (free, every CI run / dev loop):
   - New flag on tools/soak_driver.py:
     --cli-replay <path/to/cassette.jsonl>
   - Cassette format: one JSON object per line, fields:
     {recorded_ts, session_id, prompt_hash, recorded_latency_ms,
      response_json, model}
   - Replay implementation: replace the CliPool with a ReplayPool that
     yields ReplayWorker; ReplayWorker.send(prompt) sleeps
     recorded_latency_ms then returns response_json. Match by
     prompt_hash; on miss, log + return the cassette's first entry
     (best-effort) and increment a miss-counter the report surfaces.
   - Tests pool/bus/governance/HITL plumbing only — does NOT validate
     model behavior. Latency numbers from this tier are RELATIVE
     regression signals, not absolute targets.

2. RECORD-CASSETTE (weekly refresh, Haiku):
   - New tool: tools/cassette_record.py
   - Runs a full 30-min soak with
     `claude -p --model claude-haiku-4-5-20251001`
   - Writes tests/fixtures/soak_cassette_<YYYYMMDD>.jsonl
   - Cheap baseline refresh (Haiku ~10x cheaper than default model);
     catches model-side envelope drift week-over-week.
   - Symlink/pointer file tests/fixtures/soak_cassette_latest.jsonl
     points at most recent recording so CI replay always picks current.

3. SHIP-GATE SOAK (minor version cut, default model):
   - Existing tools/soak_driver.py --cli-pool-size 2 path, unchanged.
   - Source of truth for ADR-5 absolute latency numbers.
   - Required to cut a v1.x.0 tag.

Scope:

1. tools/soak_driver.py:
   - Add --cli-replay <path> flag, mutually exclusive with --cli-pool-size.
   - When --cli-replay set: instantiate ReplayPool instead of CliPool.
   - Report header records mode (replay/record/ship-gate) and cassette
     path if applicable. p95 line clarifies "RELATIVE" suffix in replay.

2. New module src/stream_manager/cli_replay_pool.py:
   - class ReplayPool: same surface as CliPool (acquire/release/shutdown).
   - class ReplayWorker: send(prompt) -> dict via cassette lookup.
   - Cassette loader: streams JSONL, builds prompt_hash → entry index.

3. New tool tools/cassette_record.py:
   - Wraps tools/soak_driver.py: forces --cli-pool-size 2,
     --model claude-haiku-4-5-20251001, --record-cassette <out_path>.
   - --record-cassette flag on soak_driver.py: tee every CLI exchange
     to the cassette JSONL alongside normal soak execution.

4. Tests tests/test_soak_replay.py:
   - Synthesize a 5-event cassette fixture.
   - Run soak_driver in --cli-replay mode against the fixture.
   - Assert: zero `claude` binary invocations (mock subprocess.Popen
     to fail loudly if called), latency within recorded ±5%, all
     governance plumbing (bus envelopes, HITL queue, registry) still
     wired correctly.

5. ADR-17 (new): docs/adr/ADR-17-soak-tiers.md — three-tier model,
   cassette format spec, "RELATIVE vs ABSOLUTE" warning. Cross-link
   from ADR-5 §"measurement methodology" so the budget doc points to
   ship-gate as the only source.

DOD:
- Replay-mode soak runs end-to-end without `claude` binary on PATH
- Cassette recorder writes a re-runnable artifact
- tests/test_soak_replay.py exercises the replay path; passes in CI
- ADR-17 merged
- ADR-5 updated with measurement-tier cross-reference
```

## TASK B — Session selector (CLI + UI) (P1)

```
End-user discoverability of which SM session/job is being monitored.
v1.1 dashboard surfaces governance state but doesn't expose a session
picker, and there is no `sm` CLI for listing/tailing sessions. Operator
running multiple SM workspaces today has to grep gov.db by hand.

Reference:
- src/stream_manager/governance.py — EngineRegistry.refresh_status
  (DO NOT touch this method's signature — Task M protected)
- src/stream_manager/message_bus.py — sessions table, envelope schema
- dashboard/server.py — /api/registry/active endpoint (Task M shipped)
- dashboard/static/ — UI bundle root

DO NOT touch (v1.1 protected surface):
- EngineRegistry.start_refresh / stop_refresh / refresh_all /
  refresh_status / last_refresh_ts / _refresh_timer — Task M
- /events SSE envelope schema (already carries session_id)
This task READS these surfaces, never modifies them.

Scope:

1. New module tools/sm_cli.py:
   - argparse entrypoint with subcommands:
     - `sm sessions list` — rows from gov.db `sessions` table joined
       with last-message ts (max(ts) per session_id from envelopes
       table) and active flag from EngineRegistry.refresh_status.
       Output table: session_id | started_at | last_msg_at | active.
     - `sm sessions tail <id>` — stream that session's bus envelopes
       from MessageBus, follow-mode (like tail -f). Exit on Ctrl-C.
   - Wraps MessageBus queries directly; reuses existing connection
     pattern from tools/sm_consumer.py.
   - pyproject.toml: register `sm` console_scripts entry point so
     `pip install -e .` exposes the binary.

2. Dashboard session picker:
   - New header component dashboard/static/session-picker.js (or
     framework-equivalent — match existing dashboard idiom).
   - Sources options from /api/registry/active (already shipped).
   - On selection change: filters all panes (decisions, HITL queue,
     latency, command stream) by selected session_id. /events SSE
     already carries session_id per envelope, so the filter is
     client-side — no server changes needed.
   - Default selection: most recently active session
     (max last_refresh_ts).
   - "Show all" pseudo-id ("__all__") preserves current behavior.
   - Persist selection to localStorage keyed on dashboard origin so
     refresh keeps the selected session.

3. Tests:
   - tests/test_sm_cli.py: list returns expected columns; tail emits
     envelopes for matching session_id and skips others; exit-code 0
     on graceful Ctrl-C simulation.
   - tests/test_dashboard_session_picker.py (live uvicorn, mirror
     test_dashboard_og7_banner.py pattern): picker dropdown
     populated from /api/registry/active; selecting an id filters
     /events SSE consumption client-side (verify via DOM-rendered
     pane content).

4. Docs:
   - README §"Operator UI" — add screenshot/description of picker.
   - REQUIREMENTS.md FR-OG-X (next available) — session picker as
     operator surface requirement.

DOD:
- `sm sessions list` exits 0 with at least the active session listed
- `sm sessions tail <id>` streams matching envelopes
- Dashboard header picker switches all panes
- Existing tests stay green (regression on /events SSE consumers)
- New tests/test_sm_cli.py + tests/test_dashboard_session_picker.py
  pass
```

## TASK C — Claude Code lifecycle bridge (P1)

```
Surface Claude Code background jobs (`BG <id>`) and `Agent(...)`
subagent spawns into SM's bus so the dashboard can show them. Today
SM only sees conversation/decision events; Claude Code's internal
process management (background bash jobs, spawned agents) is invisible.
When the user runs a background soak or spawns a subagent for a task,
the SM dashboard shows nothing — the operator can't tell which job is
being governed.

Reference:
- src/stream_manager/message_bus.py — envelope schema, bus.publish
- dashboard/server.py — /events SSE multiplexer
- (upstream) Claude Code hook stream — UserPromptSubmit / TaskOutput
  hooks; check %LOCALAPPDATA%\claude\... for hook config

DO NOT touch (v1.1 protected surface):
- MessageBus envelope schema additions must be additive (new event_type
  values; do not redefine existing ones).
- /events SSE consumer code — extend, don't replace.

Scope:

1. Hook surface investigation (do this FIRST):
   - Probe Claude Code's existing hook stream. Required event types:
     - BackgroundJobStart / BackgroundJobEnd
     - AgentSpawn / AgentComplete
   - If existing UserPromptSubmit / TaskOutput hooks don't carry
     these, two paths:
     (a) File an upstream feature request for the hook surface and
         block this task pending response.
     (b) Shim: poll Claude Code's task output directory
         (`%LOCALAPPDATA%\claude\...\tasks\<id>.output`) and synthesize
         start/end events from file appearance/last-modified.
   - Pick (b) if upstream timeline > 2 weeks; (a) preferred if
     upstream is responsive.

2. New module src/stream_manager/lifecycle_bridge.py:
   - class LifecycleBridge:
     - __init__(self, bus: MessageBus, source: 'hooks' | 'shim')
     - start() — subscribes to hook stream OR starts shim poller
     - stop() — clean shutdown
   - Emits typed envelopes into MessageBus with new event_types:
     - bg_job_start  (payload: {job_id, command, started_at})
     - bg_job_end    (payload: {job_id, exit_code, ended_at})
     - agent_spawn   (payload: {agent_id, parent_session_id,
                                 prompt_summary, spawned_at})
     - agent_done    (payload: {agent_id, status, ended_at})

3. Governance opt-in:
   - bg jobs and agents do NOT auto-route to L4 alignment.
   - Default decision = ALLOW with `track_only` flag set in the
     envelope so existing observers can ignore them.
   - Operator can wire a policy that flips track_only → governed,
     but the BRIDGE itself never spends quota.
   - Avoids quota burn from the bridge spawning its own L4 calls.

4. Dashboard panel:
   - New "Active jobs / agents" pane in dashboard, listing live
     BG job IDs + agent IDs + start ts + status.
   - Pairs with Task B session selector — selecting a session_id
     filters the panel to bg jobs / agents owned by that session
     (parent_session_id field on envelopes).
   - Closed rows fade after 60s post-completion.

5. Tests tests/test_lifecycle_bridge.py:
   - Synthesize a hook event (or fake the shim's directory state).
   - Assert: bridge emits matching envelope within 2s.
   - Assert: governance does NOT escalate the envelope (track_only
     short-circuit verified by mocking CliGovernor and asserting
     zero calls).
   - Round-trip: spawn → /events SSE delivery → dashboard DOM update
     (live uvicorn, matching test_dashboard_og7_banner pattern).

6. ADR-18 (new): "Lifecycle bridge: hook source vs shim, track_only
   default, dashboard surface". Decision rationale + revisit conditions.

DOD:
- Spawning a background bash via Claude Code or an `Agent(...)`
  subagent shows up in the dashboard within 2s
- Completion event closes the row
- No governance decision is spent on track_only events (verified by
  CliGovernor call-counter in tests)
- tests/test_lifecycle_bridge.py covers the round-trip
- ADR-18 merged
- Estimate revisit: 1–2 sessions plus 0–1 session of upstream work if
  hook surface needs an extension
```

## TASK F — Per-instance HMAC keypairs default (P3, conditional)

```
ONLY run this task if a multi-tenant deployment signal arrives. v1.0
ADR-13 locked single-shared-secret HMAC and explicitly deferred
per-instance keys. v1.1 Task O held the same conditional. v1.2 keeps
it conditional — the trigger has not arrived as of the v1.1.0 ship.

Trigger conditions (any one suffices):
- User explicitly requests multi-tenant
- Threat model changes (e.g. SM hosted as a service)
- Audit / compliance requirement surfaces
- Operator deploys SM for >1 user on shared infra

If triggered, scope is:

1. .bridge/ directory becomes per-session: .bridge/{session_id}/secret
   - File mode 0600 on POSIX; equivalent ACL on Windows.
   - Generated lazily on first session bus init.

2. desktop_commands.sign / validate accept session_id and resolve
   secret per session.
   - Cache resolved secrets in memory keyed on session_id (LRU,
     bounded ≤ 64 entries) to avoid disk I/O per call.
   - Cache invalidation on session shutdown.

3. CommandConsumer __init__ takes session_id-scoped secret resolver
   instead of a fixed shared-secret kwarg.

4. Migration:
   - Keep shared-secret path for one cycle as a fallback.
   - New sessions default to per-session key.
   - Old sessions read shared-secret if no per-session secret file
     exists. Log INFO once per session at startup so migration is
     observable.

5. ADR-19 (new): supersedes ADR-13. Document threat model delta,
   migration window, and rollback plan.

DOD:
- Two consumers with different keys cannot decrypt each other's
  commands (negative test in tests/test_hmac_isolation.py)
- Existing v1.1 deployments continue to function during migration
  window (mixed-mode test)
- Audit log shows per-session key usage
- ADR-19 supersedes ADR-13 (ADR-13 marked "superseded by ADR-19")

Skip until triggered. Do not pre-emptively implement.
```

## TASK G — Browser dashboard auth (P3, conditional)

```
ONLY run this task if a remote-access signal arrives. v1.0 and v1.1
shipped the dashboard with no auth — local-only assumption (operator
runs SM on their workstation, dashboard binds 127.0.0.1). The first
time anyone exposes the dashboard beyond loopback, this task gates.

Trigger conditions (any one suffices):
- User asks to expose dashboard on a non-loopback interface
- Reverse-proxy / tunnel deployment requested
- Browser dashboard accessed from a machine other than the SM host
- Audit / compliance flags the unauth surface

If triggered, scope is:

1. Auth scheme decision:
   - Default: shared-secret bearer token, generated on first
     dashboard boot, stored in .bridge/dashboard.token (mode 0600).
   - Operator copies token to the browser via a one-time
     localhost-only setup page; subsequent requests carry it as a
     cookie.
   - Stretch: OAuth/SSO via existing project IdP — defer to v1.3
     unless operator explicitly asks.

2. Server side:
   - dashboard/server.py: middleware that rejects all routes except
     /healthz and /setup with 401 unless the bearer cookie matches.
   - /api/* routes covered.
   - /events SSE: cookie auth on connect.

3. Client side:
   - Setup page UX: paste token, store in cookie, redirect to /.
   - Logout endpoint clears cookie.

4. Integration with Task B session picker: ensure picker continues
   to work post-auth (cookie carried on /events SSE).

5. ADR-20 (new): "Dashboard auth: bearer cookie, opt-in by deployment
   posture". Document rotation policy + revocation steps.

DOD:
- Curling /api/* without the cookie returns 401
- Setup page works in a fresh browser
- Existing local-loopback workflow opt-in only (auth disabled by
  default if dashboard binds 127.0.0.1; auto-enabled on non-loopback
  bind)
- ADR-20 merged

Skip until triggered. Do not pre-emptively implement.
```
