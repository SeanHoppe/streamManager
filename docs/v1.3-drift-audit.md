## 2. Gap narratives

### 2.1 GAP — HITL pre-fill column `hitl_pending.bias_hint` is unread by the dashboard (anticipated by plan §10)

**What shipped.** P5d Fix B added a `bias_hint` column to `hitl_pending` so the operator-facing HITL prompt can pre-fill the Learn Mode advisory hint when one was attached at queueing time. The writer chain is intact:

- `governance.py` calls `HitlQueue.route(bias_hint=hint, ...)` when escalating.
- `hitl.py:196` accepts `bias_hint`, encodes it via `_encode_bias_hint` (`hitl.py:102-127`), passes the JSON to `MessageBus.add_hitl_pending(bias_hint=...)`.
- `message_bus.py:461-488` writes the column. Schema migration at `:297` is in place. The bus's own `list_hitl_pending` and `get_hitl_pending` SELECTs DO project `bias_hint` (`message_bus.py:520`, `:540`, `:551`, `:567`).

**What is missing.** The dashboard endpoint `dashboard/server.py:658 api_hitl_pending` is the path the HITL UI consumes (verified at `dashboard/static/index.html:2584` — `fetch('/api/hitl/pending')`). Its SELECT lists the columns explicitly:

```
SELECT hp.id, hp.message_id, hp.proposed_action,
       hp.proposed_confidence, hp.trigger_reason, hp.queued_at,
       m.session_id, m.content
FROM hitl_pending hp JOIN messages m ON hp.message_id=m.id
WHERE hp.resolved_at IS NULL ...
```

`hp.bias_hint` is not in the projection. The HTML (`dashboard/static/index.html`) contains zero references to `bias_hint` or `biasHint`. The HITL pane therefore cannot surface the Learn Mode hint to the operator at decision time. The operator chooses ALLOW/GUIDE/SUGGEST/INTERVENE/BLOCK without seeing the advisory.

**Why this matches the P5e drift pattern.** Same shape as the P5e dead-code finding — a writer (`HitlQueue.route(bias_hint=...)`) and a column (`hitl_pending.bias_hint`) shipped without verifying the operator-visible reader exists. The bus's row helpers project the column, which masks the gap from a "is the column read?" grep at the bus layer; it is the dashboard endpoint's narrower projection that drops it on the floor.

**Severity.** Medium. The bias_hint column is non-empty in production (HitlQueue is populating it), and the operator UX silently behaves as if the hint were never attached. No safety impact (P5d safety-priority short-circuit means destructive content never carries a bias hint into HITL anyway). No latency impact. Spec-vs-impl drift against P5d Fix B's stated purpose: "pre-fill the HITL pane with the Learn Mode hint."

**Recommended follow-up.** **Phase C10** — Wire dashboard HITL pane to `bias_hint`.

Scope sketch (to be detailed in a separate phase frame):

1. Extend `dashboard/server.py:api_hitl_pending` SELECT to include `hp.bias_hint`.
2. Decide JSON-decoded shape exposed to the client (decode the JSON server-side and emit object, vs. pass-through string the client decodes).
3. Update `dashboard/static/index.html` `renderHitlPanel` to render the hint inline on each pending row — at minimum `category` + `confidence` + `ladder_step_suggestion`. Visual treatment: subordinate to the action buttons; clearly marked "advisory."
4. Test: extend `tests/test_advisory_bias.py` (or a new `tests/test_dashboard_hitl_bias_hint.py`) to assert the API response includes `bias_hint` when the row was queued with one and an empty/absent value when not.
5. Do-not-touch guard: `_encode_bias_hint`, the `hitl_pending` schema, the safety-priority short-circuit gate.

## 3. Non-gaps (audit clean)

### 3.1 JSONL tail (P5b) — audit clean

`src/stream_manager/jsonl_tail.py:312` and `:349` publish `desktop_prompt`/`user_reply` with `metadata.pair_id`. `src/stream_manager/learn_categorizer.py:517-525` performs the documented `messages u JOIN messages d ON d.id = json_extract(u.metadata, '$.pair_id')` query and filters on `u.type='user_reply' AND d.type='desktop_prompt'`. Wire is real and exercised end-to-end by `tests/test_jsonl_tail_learn_mode.py`.

### 3.2 Categorizer audit log (`learn_patterns`) — audit clean (under ADR-19)

`learn_patterns` is the append-only audit log. Pre-C1, `bias_for` reads it directly (`learn_categorizer.py:739-750`). Post-C1, the reader migrates to `learn_patterns_canonical` and the audit log becomes a write-only forensic record. This is intentional under the canonical/audit split (ADR-19, filed in C6). The audit log having no live production reader after C1 is a documented design property, not a drift gap.

### 3.3 Bias-applied audit envelope (P5d) — audit clean (with cosmetic follow-up)

`src/stream_manager/governance.py:704-743` emits the `learn_mode_bias_applied` envelope. The dashboard SSE forwarder (`dashboard/server.py:2079-2117`) ships ALL `messages` rows as `event_type` SSE frames. The HTML event-log (`dashboard/static/index.html:3490-3501` via `appendEvlogRow`) renders any unknown `event_type` with the generic `evt-default` class. So the operator does see the envelope land — it just isn't styled or labelled like the explicitly-known types in `_KNOWN_EVT_CLASSES`.

This is a genuine wire (the envelope reaches the operator surface). The cosmetic gap — no dedicated styling/label — is recommended as **C9** (cosmetic only, not load-bearing for v1.3 ship-gate).

### 3.4 Decay → bias seam (P5e) — already filed as C1

The original triggering finding. Pre-C1: `decay_sweep`/`consolidate_patterns` write `learn_patterns_canonical` (`decay.py:159`, `:172`, `:183`, `:195`, `:270`); `bias_for` reads `learn_patterns` (`learn_categorizer.py:745`). Post-C1: `bias_for` migrates. Already covered by the corrective plan.
