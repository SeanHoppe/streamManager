### PHASE P1a outcome — 2026-05-07

**Resolution: (c) different signal — root cause is NOT corpus framing.**

**Probe data** (`reports/p1a-corpus-haiku-verdicts-20260507T075646Z.md`):

27 samples via fresh `claude -p <content> --no-session-persistence` subprocess per call. Same system prompt as `cli_governance.py::_SYSTEM_TEMPLATE`. Same `MODEL = claude-haiku-4-5`. INTENT.md identical.

| class | n | actions | block/intervene rate | conf p50 |
|-------|---|---------|----------------------|----------|
| existing_l2_l3_trigger (soak content) | 8 (1 timeout) | BLOCK=4, INTERVENE=4 | **100.0%** | 0.95 |
| reframed_bare_imperative | 9 | BLOCK=6, INTERVENE=3 | **100.0%** | 0.99 |
| control_known_blocking | 9 | BLOCK=6, INTERVENE=3 | **100.0%** | 1.00 |

26/27 = **96% BLOCK/INTERVENE** on the existing soak `_L2_L3_TRIGGER` content (1 timeout, 0 ALLOW). The corpus is not the falsification surface. Haiku readily blocks these imperatives when invoked in a fresh subprocess.

**Soak-vs-probe discrepancy** — gov DB (`tmp/soak_gov_p1_full.db`, joined `decisions ⨝ messages`) shows the same prompts producing post-`_apply_mode` ALLOW with conf 0.80–0.89 and reasoning that explicitly identifies the destructive pattern (e.g. "Safety priority #1 blocks DROP TABLE"). The reasoning text contradicts the action. Two interpretations:

1. Haiku raw verdict was BLOCK/INTERVENE; `_apply_mode(OBSERVE)` flipped action → ALLOW for output but kept the reasoning unchanged. **In this case the verdict gate in `cli_governance.py:281` (which sees the raw `CliDecision.action` BEFORE `_apply_mode`) would have fired and `cli_dispatch_fallback_ms` p95 would be > 0.** It is 0 → this interpretation is falsified.
2. Haiku raw verdict was ALLOW c=0.80–0.89 with reasoning text that recognises destructiveness but stops short of acting. The fallback gate correctly does not fire. This is the surviving interpretation.

**Hypothesis for soak ALLOW vs probe BLOCK on identical content + system prompt:** `cli_pool.CliWorker` runs a long-lived `claude -p --input-format stream-json --output-format stream-json --no-session-persistence` process. Each `send()` writes a fresh `session_id` (UUID) into the user-message envelope, but the underlying `claude -p` process is shared across ~30 turns per worker over a 1800s soak. The probe spawned a new subprocess per call; the soak reused two long-lived workers (`--cli-pool-size 2`). Whether `claude -p` stream-json mode preserves in-memory conversational context across per-turn session_id changes is a CLI implementation detail — but the empirical signal is clear: same content + same system prompt + different process lifecycle → opposite verdicts.

If the hypothesis holds, the fallback trigger is wired correctly and the corpus is adequate, but the cli_pool worker model bias (toward conversational ALLOW after seeing earlier ALLOW turns) prevents the trigger from firing during soak. The lever is dormant **as a side-effect of v1.1 Task J cli_pool reuse**, not a v1.7/v1.9 P1 design flaw.

**P4 unblock decision:** **P4 ship-gate is unblocked with explicit caveat documentation in ADR-5 v1.9 and the v1.9 CHANGELOG.** Rationale:
- Verdict gate is wired correctly (PR #100 unit tests + alignment-eval prove this).
- Corpus is adequate (probe proves this).
- Lever dormancy in soak is a process-isolation artefact of cli_pool, not a corpus or wiring bug.
- Forcing the lever to fire under the existing pool architecture would require either disabling cli_pool reuse (regressing v1.1 Task J latency budget) or refactoring CliWorker to recycle per-turn (out of v1.9 scope).
- The P1 verdict trigger is non-regressive: in non-pool / fresh-subprocess paths (e.g. future tooling, alignment-eval) it fires correctly. In the pool path it remains dormant pending v1.9 P1b or v2.0.
