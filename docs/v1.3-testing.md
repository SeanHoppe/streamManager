# v1.3 testing methodology

**Companion docs:**
- `docs/v1.3-task-plan.md` — phase plan; references this file from P1, P2, P5
- `docs/v1.4-backlog.md` — testing scripts/scenarios library buildout (deferred)

**Premise.** SM proactively instructs the desktop operator to perform a predetermined action and asserts the expected envelope sequence/category. Testing then has a deterministic operator-in-the-loop driver instead of relying on organic Desktop traffic. The harness times out gracefully if the operator skips, and SM never auto-fires Desktop prompts — the operator confirms each step.

This doc defines three methods. P1 and P2 use Method 1. P5e uses Methods 2 and 3. Library buildout for the v1.0–v1.3 surface is deferred to v1.4 (see `docs/v1.4-backlog.md`).

---

## Operator-in-the-loop guard rails (apply to all three methods)

- SM never auto-fires Desktop prompts. Each scripted step prints to stdout and waits for operator confirmation (Enter, or copy-paste of a sentinel string back into the harness).
- Operator confirms each step. The harness blocks until either the expected envelope arrives, the operator skips (sentinel), or the timeout fires.
- Harness times out gracefully — emits a `SKIPPED` row in the report, does not hang.
- All three artifact directories (`tests/scenarios/`, `tests/beacons/`, `tests/probes/`) are committed to the repo so a fresh contributor can run them without ad-hoc Desktop traffic.

## Method 1 — Scripted Scenario Replay

**Premise.** A YAML script enumerates `(operator_prompt, expected_envelopes, timeout_s)` tuples. The driver prints each prompt to stdout (operator copy-pastes into Desktop) then waits on the SM bus for matching envelope sequence within the timeout.

**Artifact location:** `tests/scenarios/<name>.yaml`

**Artifact format:**
```yaml
name: <scenario-name>
description: <one-line>
steps:
  - operator_prompt: "Run the following in Desktop: <verbatim text>"
    expected_envelopes:
      - type: <envelope_type>
        payload_match:
          <field>: <expected_value_or_regex>
    timeout_s: 30
  - operator_prompt: "..."
    expected_envelopes: [...]
    timeout_s: 30
```

**Harness invocation:**
```
python tools/scenario_runner.py tests/scenarios/<name>.yaml
```

Exit code 0 = pass, 1 = mismatch (with diff dump showing observed vs expected envelope sequence).

**Pass/fail criteria:**
- Pass: envelope order matches, envelope types match, `payload_match` fields equal (or regex-match) observed payloads, all within `timeout_s`.
- Fail: envelope missing, out-of-order, type mismatch, payload mismatch, or operator skipped without confirming. Driver dumps observed vs expected diff to stderr.

**Which phases use it:**
- P1 — Lifecycle bridge regression (driver/recorder hardening tests)
- P2 — `list_active_jobs` correctness (overflow scenario)

**Worked example (`tests/scenarios/list_active_jobs_overflow.yaml`):**
```yaml
name: list_active_jobs_overflow
description: Verify list_active_jobs does not truncate at 25+ pairs
steps:
  - operator_prompt: |
      In Desktop, spawn 30 background bash jobs:
        for i in {1..30}; do (sleep 60 &); done
    expected_envelopes:
      - type: bg_job_start
        payload_match:
          job_id: ".+"
    timeout_s: 10
  - operator_prompt: "Hit /api/lifecycle/jobs and confirm 30 rows returned"
    expected_envelopes:
      - type: lifecycle_query_result
        payload_match:
          row_count: 30
    timeout_s: 5
```

## Method 2 — Expectation Beacon (HITL-gated)

**Premise.** SM emits a `test_beacon` envelope into the bus before each step (e.g. "expect category=git_force_push within 30s"). Operator triggers Desktop with a predetermined phrase. SM auto-asserts that the next incoming envelope matches the beacon assertion. The dashboard pane shows green/red per beacon for live observation.

**Artifact location:** `tests/beacons/<feature>.jsonl`

**Artifact format (one JSON object per line):**
```jsonl
{"beacon_id": "lm-001", "operator_prompt": "Tell Desktop: 'force push to main'", "assertion": {"category": "git_force_push", "confidence_min": 0.7}, "timeout_s": 30}
{"beacon_id": "lm-002", "operator_prompt": "Tell Desktop: 'list git branches'", "assertion": {"category": "git_read_only"}, "timeout_s": 30}
```

**Harness invocation:**
```
python tools/scenario_runner.py --beacons tests/beacons/<feature>.jsonl
```

Driver emits a `test_beacon` envelope per row, prints `operator_prompt`, waits on the bus for the next envelope of the predicted type, asserts `assertion` fields match. Each row reported PASS / FAIL / SKIPPED on stdout and as a green/red row in the dashboard beacon pane.

**Pass/fail criteria:**
- Pass: beacon resolved within timeout AND incoming envelope payload matches all `assertion` fields.
- Fail: timeout fired, OR resolved envelope payload mismatched assertion.
- Skipped: operator emitted skip sentinel — counted separately, does not fail the run.

**Which phases use it:**
- P5e — Learn Mode categorizer (beacon → expected category prediction)

**Worked example (`tests/beacons/learn_mode_categorizer.jsonl`):**
```jsonl
{"beacon_id": "lm-cat-001", "operator_prompt": "In Desktop type: 'please force push my fix to main'", "assertion": {"category": "git_force_push", "confidence_min": 0.7}, "timeout_s": 30}
```

## Method 3 — Adversarial Drift Probe

**Premise.** A CSV table of near-miss prompts (similar wording, different intent). Operator sends each prompt back-to-back. Driver asserts NO false-positive promotion in the `learn_patterns` ladder, AND asserts contradiction snap-demote logic fires on rows whose `true_category != distractor_categories` family.

**Artifact location:** `tests/probes/<name>.csv`

**Artifact format:**
```csv
prompt,true_category,distractor_categories
"force push my fix to main",git_force_push,"git_safe_push,git_pull_request"
"force a rebuild of the docker image",docker_rebuild,"git_force_push,docker_pull"
"push the new tag to origin",git_tag_push,"git_force_push,git_safe_push"
```

**Harness invocation:**
```
python tools/scenario_runner.py --probes tests/probes/<name>.csv
```

Driver iterates rows, prompts operator to send each, observes the `learn_patterns` table state after each, asserts:
1. Top-N pattern for the prompt matches `true_category` (not any of `distractor_categories`).
2. Patterns categorized into a `distractor_category` from a prior row do not get promoted by the current row.
3. When a row's true category contradicts a previously-promoted pattern, contradiction snap-demote fires per the design ladder.

**Pass/fail criteria:**
- Pass: zero false promotions across the table; all expected snap-demotes observed at design threshold.
- Fail: ANY false promotion (a `distractor_category` pattern's `ladder_step` increased), OR a snap-demote that should have fired did not.

**Which phase uses it:**
- P5e — Learn Mode decay + safety-priority preservation

**Worked example (one row from `tests/probes/learn_mode_drift.csv`):**

`prompt = "force push my fix to main"`, `true_category = git_force_push`, `distractor_categories = git_safe_push, git_pull_request`. After operator sends the prompt, driver asserts:
- A `git_force_push` pattern row exists in `learn_patterns` for this prompt's hash.
- No `git_safe_push` or `git_pull_request` pattern row's `ladder_step` increased as a result of this prompt.
- If a prior row had promoted a `git_safe_push` pattern that overlaps this prompt's hash, `contradicted_count` increments AND `ladder_step` snap-demotes per the design threshold.

## Summary table

| Method | Artifact | Driver | Best for | Phase |
|---|---|---|---|---|
| 1 — Scripted Scenario Replay | `tests/scenarios/*.yaml` | `tools/scenario_runner.py <path>` | Lifecycle/query regression | P1, P2 |
| 2 — Expectation Beacon | `tests/beacons/*.jsonl` | `tools/scenario_runner.py --beacons <path>` | Categorizer prediction | P5e |
| 3 — Adversarial Drift Probe | `tests/probes/*.csv` | `tools/scenario_runner.py --probes <path>` | Decay + false-positive guard | P5e |

The library buildout (a comprehensive set of canned operator scripts covering the v1.0–v1.3 surface) is tracked in `docs/v1.4-backlog.md`. Goal: a fresh contributor can run `python tools/scenario_runner.py --all` and validate the entire SM surface without ad-hoc Desktop traffic.
