# v1.3 corrective action plan

**Trigger:** PR #64 (P5e — decay + canonical projection) review found that `learn_patterns_canonical` is written by `consolidate_patterns` and `decay_sweep` but never read by `bias_for`. The verdict path still queries `learn_patterns` (audit log). Decay/reinforcement/contradiction work is computed and silently discarded.

**Status:** Active. Blocks the P5 close-out PR (`ship/v1.3-learn-mode` → `main`).

**Owner:** SeanHoppe.
**Started:** 2026-05-04.
**Target close:** before P6 ship-gate finalize.

---

## 1. Incident summary

### 1.1 What shipped
- P5c (PR #62) created `learn_patterns` (append-only audit log). `bias_for` (P5d) reads it via `ORDER BY last_reinforced_ts DESC`.
- P5e (PR #64) created `learn_patterns_canonical` (UPSERT, `UNIQUE(prompt_hash)`). `consolidate_patterns` and `decay_sweep` write only here.
- No reader was migrated from `learn_patterns` to `learn_patterns_canonical`.

### 1.2 What design promised
`docs/learn-mode-design.md` §3.2 lists a single "patterns" concept consumed by the bias reader. The decay scheduler updates ladder rung; the bias reader reads ladder rung. FR-LM-4 ("Decay ladder + reinforcement reset + contradiction snap-demote") implies the decay-modified state reaches the verdict path.

### 1.3 Effective state in main-tracking
- `BiasHint.ladder_step_suggestion` is always 0 (audit-log rows are inserted with `ladder_step=0` by P5c and never updated).
- `last_reinforced_ts` reflects "newest insert" not "newest reinforcement."
- Aging via `decay_sweep` is invisible to all consumers except `decay.py` itself, the test suite, and the scenario probe.
- Append-only "newest wins" gives a poor man's reinforcement that works without P5e at all.

### 1.4 Severity
- Spec violation against FR-LM-3 and FR-LM-4.
- Marquee feature surface (operator-visible behaviour) unchanged: same audit envelopes fire.
- Latency: unchanged (no hot-path consumer to slow down).
- Safety priorities: untouched (P5d safety-priority short-circuit is independent of canonical/audit split).

The deferral was not authorized by P5a design. | Subagent escape hatches that contradict the design doc were not flagged at review time. |

The corrective plan addresses every row above. Without that, the next cycle drifts on a different seam.

## 12. PHASE C6 — ADR-19 Learn Mode canonical/audit split

**Pain point:** P5e introduced the canonical/audit split inside the cycle without an ADR. Future maintainers see two tables doing similar work and either re-merge them or invent a third. The split is load-bearing for FR-LM-4 (UPSERT semantics on `prompt_hash`) but the rationale is undocumented outside this conversation.

**References:**
- `docs/learn-mode-design.md` §3.2 (single-table model).
- PR #64 schema comment in `src/stream_manager/message_bus.py`.
- This document §1.

**Do-not-touch guard:** ADRs once shipped are historical record. ADR-19 ships once, in this phase, and is not edited later (status updates only via successor ADRs).

**Scope:**
1. New `docs/adr/ADR-19-learn-patterns-canonical-split.md`:
   - **Status:** Accepted.
   - **Date:** 2026-05-04.
   - **Context:** P5c shipped `learn_patterns` as append-only audit log. P5c review nit asked for `UNIQUE(prompt_hash)`. P5e responded with a separate `learn_patterns_canonical` table to preserve audit history while exposing UPSERT semantics for decay/reinforcement.
   - **Decision:** Two-table split kept as v1.3 contract. Audit log = source of truth for "what categorizations did we ever observe." Canonical = current effective bias for each `prompt_hash`. `bias_for` reads canonical (post C1).
   - **Consequences:** `consolidate_patterns` is the only writer to canonical. Audit log writes are independent (P5c categorizer). Storage cost is modest (one canonical row per unique prompt_hash). A future cycle may collapse to a single table with a `current=1` flag if audit-log retention requirements change.
   - **Alternatives considered:** (a) single table with `UNIQUE(prompt_hash)` + UPSERT, losing audit history; (b) materialized view; (c) keep audit-only and discard P5e — rejected because FR-LM-4 requires explicit decay state.
2. Cross-link from `docs/learn-mode-design.md` §3.2 component table.
3. Cross-link from `REQUIREMENTS.md` §4.12 FR-LM-4.

## 16. Drift-prevention process patches (apply going forward)

These are the durable changes — kept in memory so v1.4 inherits them.

1. **Cross-PR seam review at sub-cycle close-out.** Before any feature branch (e.g. `ship/v1.X-feature`) merges to `main`, run a writer↔reader audit against the design doc's component table. This is C4's pattern, made permanent.
2. **End-to-end pipeline test as a sub-cycle DOD.** Every multi-PR feature ships a single test that exercises every sub-PR's contribution in one process. Sub-cycles that lack this test do not pass close-out.
3. **FR-numbered tests for marquee features.** Anything that ships a new `FR-*` entry in `REQUIREMENTS.md` ships a `test_fr_<area>_<n>_*` test naming convention so future grep finds the enforcement point.
4. **Subagent escape-hatch flag.** Any subagent return containing "deferred to a follow-up," "the X reader is unchanged," or equivalent triggers a manual re-review against the design doc before merge.
5. **ADR-or-it-didn't-happen for in-cycle architecture changes.** Any sub-phase that introduces a new table, new module-level abstraction, or new component layout that was not in the cycle's design doc must ship an ADR in the same PR.
