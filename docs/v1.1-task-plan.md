## TASK I — Hydrator hot-path profile + lazy-init (P0)

```
Profile and fix the v1.0 p95 latency regression. Soak baseline v1.0 final
shows p95 = 19.08s vs ADR-5 budget 15s (28% over). Suspect: cross-session
Hydrator thread spawned during EngineRegistry.get_or_create — it reads
the patterns table (cross_session=1 rows) and injects each into the
engine's DecisionGraph at L1 advisory level.

Step 1 — measurement before refactor:

1. Add a per-call timing harness to tools/soak_driver.py (or a dedicated
   tools/perf_probe.py) that records, per evaluate() call:
   - wall-clock total
   - time spent inside engine construction (registry.get_or_create)
   - time spent in Hydrator.run (if it ran inline that call)
   - time spent in CLI subprocess (existing instrumentation)
2. Run a 5-min focused probe with cold cache (delete tmp/soak_gov.db
   before run) AND with warm cache (re-run preserving DB).
3. Write reports/perf-hydrator-{ISO-ts}.md with per-call breakdown.

Step 2 — fix (only if profile confirms Hydrator is the cause):

Move Hydrator off the hot path. Two designs to choose from based on
profile data:

A) Lazy-init: do not spawn Hydrator in EngineRegistry.get_or_create.
   Spawn on first evaluate() call after engine creation, in a daemon
   thread. evaluate() does not block on Hydrator completion (already
   the case per Task F design — engine.hydrated flag is informational).
   Net: first evaluate() is unaffected because Hydrator runs concurrent;
   but engine construction is freed of any sync DB read.

B) Pre-spawn pool: EngineRegistry maintains a small pool of
   "pre-hydrated" engines (with the cross-session graph already
   injected). get_or_create assigns a pool engine to a session_id and
   re-binds session_id, bus, hitl, etc. Engine construction becomes
   essentially free.

Pick A unless profile shows (A) doesn't move p95 enough. (B) has higher
complexity and is only justified by data.

Step 3 — verify:

- Re-run 30-min soak post-fix. Target: p95 ≤ 15s (back inside ADR-5
  budget). Stretch: p95 ≤ 10s.
- New tests: tests/test_hydrator_lazy_init.py — assert that
  EngineRegistry.get_or_create does NOT block on Hydrator completion;
  assert engine.hydrated transitions False → True after a bounded time.

DOD:
- reports/perf-hydrator-* report committed
- p95 ≤ 15s in fresh 30-min soak
- New tests pass
- ADR-5 latency-budget doc updated if v1.1 target ratchets down
```

## TASK J — CLI subprocess warm-pool (P0)

```
Drop p50 governance latency from 6.4s to ~2s by reusing CLI subprocesses
across calls. Today, every governance escalation spawns a fresh `claude`
subprocess; the cold-start dominates wall-clock per ADR-5 rationale.

Reference:
- src/stream_manager/cli_governance.py — current spawn-per-call path
- src/stream_manager/cli_client.py — subprocess wrapping
- ADR-5 v1.0 budget; v1.1 target p50 ≤ 3s

Scope:

1. New module src/stream_manager/cli_pool.py:
   - class CliPool:
     - __init__(self, size: int = 2, model: str | None = None)
     - acquire() -> CliWorker (context manager-friendly)
     - release(worker: CliWorker)
     - shutdown() -> None
   - CliWorker wraps a long-lived `claude` process with stdin/stdout
     pipe. Sends one prompt, reads one JSON response, returns.
   - Process recycle: after N calls (default 50) or on any decode
     error, kill + respawn so we never accumulate state in a worker.
   - Health probe at acquire-time: send "{}" → expect non-empty
     response within 2s; if no response, kill + respawn.

2. Wire into cli_governance.CliGovernor:
   - Constructor accepts an optional pool kwarg. When supplied, route
     through pool.acquire() instead of spawning a new subprocess.
   - When pool is None (default), preserve existing spawn-per-call
     path so existing tests don't break.

3. Lifecycle in dashboard server / soak driver:
   - dashboard/server.py builds a CliPool at startup, passes to
     CliGovernor (and EngineRegistry if it forwards).
   - Register atexit + SIGTERM handlers so workers shut down cleanly.
   - PID file at .bridge/cli-pool.pids — boot-time reaper kills any
     stale workers from previous run.

4. Tests:
   - tests/test_cli_pool.py — pool acquire/release ordering, health
     probe, recycle after N calls, shutdown kills all workers.
   - Skip on CI / no-CLI environments via a marker that checks for
     the `claude` binary on PATH (existing pattern in soak_driver.py).

5. ADR-5 update: revise budget to p50 ≤ 3s, p95 ≤ 8s if soak
   confirms. Otherwise document measured numbers and re-baseline.

DOD:
- pytest green (CLI-bearing tests skipped in non-CLI env)
- 5-min focused soak with pool: p50 ≤ 3s
- 30-min soak still PASS, no orphan process leaks (psutil scan post-run)
- Stale-worker reaper verified by manual SIGKILL the dashboard mid-run,
  restart, confirm no leftover claude.exe processes
```

## TASK K — desktop_command long-poll → SSE (P1)

```
OQ4 v1.0 lock chose long-poll for transport. v1.1 swaps to SSE for
sub-second command delivery. Producer keeps writing to the same
desktop_commands WAL table; consumer subscribes to a new SSE endpoint
that streams new pending rows on insert (via existing MessageBus
subscriber callback or polling-based fallback).

Reference:
- src/stream_manager/desktop_commands.py — producer (unchanged)
- src/stream_manager/desktop_command_consumer.py — current poll loop
- dashboard/server.py — /api/commands/pending endpoint (kept)
- tools/sm_consumer.py — daemon entry

Scope:

1. dashboard/server.py:
   - New endpoint GET /api/commands/stream?session_id=X (SSE,
     text/event-stream).
   - Filter rows server-side: pending status, session_id match,
     session_id != SM_OWN_SESSION_ID.
   - On connect, replay current pending rows (oldest first), then
     tail new rows via 200ms poll OR a bus.subscribe callback if
     the producer emits a bus event on insert.
   - TTL behavior preserved: rows older than 30s flip to expired
     and are NOT replayed.
   - /api/commands/pending stays — old consumers must keep working
     for one minor cycle.

2. desktop_command_consumer.py:
   - New transport mode: CommandConsumer(transport='sse'|'long-poll',
     default 'long-poll' until v1.2 default flip).
   - SSE path uses httpx.Client.stream("GET", "/api/commands/stream")
     and handles reconnect with exponential backoff (cap 10s).
   - Validate signature → execute → ack POST is unchanged.

3. tools/sm_consumer.py: --transport flag, default long-poll.

4. Tests tests/test_desktop_command_sse.py:
   - Live uvicorn (mirror tests/test_dashboard_og7_banner.py pattern)
   - producer emits → consumer SSE receives within 250ms
   - reconnect after disconnect resumes from cursor
   - SM_OWN_SESSION_ID rows still server-side filtered

5. ADR-14 (new): "desktop_command transport: SSE in v1.1, long-poll
   deprecated v1.2". Decision + rationale + migration path.

DOD:
- pytest green (long-poll path tests still pass)
- Manual: SSE consumer receives ack within 250ms of POST /api/commands
- /api/commands/pending kept (regression-tested) for one cycle
- New ADR-14 in docs/adr/
```

## TASK L — cross_session HITL hash column (P2)

```
Hygiene: Task F encoded the pattern hash inside proposed_action via the
string `flag_cross_session:{hash}` because hitl_pending had no spare
column. Cleaner: dedicated nullable column hitl_pending.matched_hash,
backfill from existing rows on migration.

Reference:
- src/stream_manager/message_bus.py — hitl_pending CREATE in _SCHEMA
- src/stream_manager/hitl.py — dispatch_resolution split-on-colon
- src/stream_manager/governance.py — observe_for_learning emit site

Scope:

1. message_bus.py _SCHEMA / migration:
   - ALTER TABLE hitl_pending ADD COLUMN matched_hash TEXT NOT NULL
     DEFAULT '' (additive, idempotent via contextlib.suppress per
     existing pattern at message_bus.py:182).
   - One-time backfill on _init_schema: for any row whose
     proposed_action LIKE 'flag_cross_session:%', set matched_hash
     to the suffix and reset proposed_action to 'flag'. Idempotent
     (re-run safe — backfill skips rows where matched_hash != '').

2. hitl.py:
   - dispatch_resolution reads hash from row.matched_hash, NOT from
     proposed_action split.
   - Keep the legacy split path for one cycle as a fallback if
     matched_hash is empty AND proposed_action contains a colon —
     covers DBs that haven't been touched since v1.0 ship.

3. governance.py:
   - queue_hitl call uses proposed_action='flag', matched_hash=hash.
     (queue_hitl signature must accept matched_hash kwarg — likely
     already there since the column exists for other rows; verify.)

4. Tests:
   - tests/test_hitl_hash_column.py: column present after migration,
     backfill correctness, both new and legacy rows resolve correctly.
   - Existing tests/test_cross_session_learning.py still passes.

DOD:
- pytest green
- sqlite3 :memory: + MessageBus init shows matched_hash column
- Backfill verified on a fixture DB seeded with legacy rows
```

## TASK M — EngineRegistry refresh wiring (P2)

Task F added EngineRegistry.refresh_all() / start_refresh() but auto_refresh
defaults to False. No production caller wires the timer. Wire it up
from dashboard/server.py boot AND tools/sm_consumer.py (or wherever
the long-lived SM daemon lives) so cross-session pattern flips picked
up by the operator UI propagate to running engines within 60s.

Reference:
- src/stream_manager/governance.py — EngineRegistry refresh API
- dashboard/server.py — _get_engine_registry lazy singleton
- src/stream_manager/cross_session_hydrator.py — Hydrator class

Scope:

1. dashboard/server.py:
   - In FastAPI startup hook, call _get_engine_registry().start_refresh().
   - In shutdown hook, call stop_refresh() and join the timer thread.
   - Expose refresh status on /api/registry/active: include
     {refresh_active: bool, last_refresh_ts: float | None}.

2. EngineRegistry:
   - Add last_refresh_ts attribute, set on each refresh_all() return.
   - Ensure refresh thread is daemonized so process exit doesn't hang.

3. Tests:
   - tests/test_engine_registry_refresh.py: start_refresh + stop_refresh
     lifecycle, last_refresh_ts increments, idempotent stop.

4. (If Task I lazy-init lands first) Coordinate: refresh_all should NOT
   re-spawn the per-engine Hydrator if it's still running for that
   engine. Use the engine.hydrated flag as a guard.

DOD:
- pytest green
- Manual: flip a pattern via dashboard "demote" → ≤60s later, a hot
  engine in the registry reflects the demoted state on next evaluate
- /api/registry/active surfaces refresh status

## TASK N — WireCLI re-incorporation (P3)

```
Bring back the WireCLI work from spike-c-final that was sidelined
during the api_governance → cli_governance migration. WireCLI provides
a structured RPC layer over the CLI subprocess (vs the current ad-hoc
JSON prompt/response). Reduces parser-fragility errors that currently
fall through to the "I cannot evaluate this" degradation path.

Reference:
- spike-c-final branch on origin (verify it still exists; if pruned,
  recover from git reflog or local backup).
- docs/wirecli-spec.md if present (find via git log on spike-c-final)
- src/stream_manager/cli_client.py — current ad-hoc parser
- src/stream_manager/cli_governance.py — escalation path

Scope:

1. Recover spike-c-final code:
   - git fetch origin spike-c-final OR git checkout spike-c-final from
     local refs. If gone, reconstruct from soak / replay logs that
     captured WireCLI traffic (check reports/replay_*).
2. Cherry-pick or port WireCLI module into src/stream_manager/wirecli.py
   (rename from spike's namespace if it was experimental).
3. Wire into CliClient as an optional transport: CliClient(transport=
   "wirecli"|"json", default "json" for one cycle).
4. Tests/test_wirecli.py: structured request/response round-trip, error
   surface (e.g. malformed response → typed exception, not silent
   ALLOW degrade), schema versioning.
5. Soak comparison: 5-min run with json transport, 5-min with wirecli.
   Document parser-fragility-induced ALLOWs in each. Goal: WireCLI
   eliminates the silent-degrade fallback.

DOD:
- pytest green
- Soak with WireCLI: zero "inner JSON parse failed" log entries
- ADR-15 (new): "WireCLI as default CLI transport in v1.2"
```

## TASK O — Per-instance HMAC keypairs (P4, conditional)

```
ONLY run this task if a multi-tenant deployment signal arrives
(operator deploys SM for >1 user on shared infra). v1.0 ADR-13
locked single-shared-secret HMAC and explicitly deferred per-instance
keys.

Trigger conditions:
- User explicitly requests multi-tenant
- Threat model changes (e.g. SM hosted as a service)
- Audit / compliance requirement surfaces

If triggered, scope is:

1. .bridge/ directory becomes per-session: .bridge/{session_id}/secret
2. desktop_commands.sign / validate accept session_id and resolve
   secret per session (cache to avoid disk I/O per call).
3. Consumer's CommandConsumer __init__ takes session_id-scoped secret.
4. Migration: keep shared-secret path for one cycle; new sessions
   default to per-session key, old sessions read shared-secret.
5. ADR-16 supersedes ADR-13.

DOD:
- Two consumers with different keys cannot decrypt each other's commands
- Existing v1.0 deployments continue to function during migration
- Audit log shows per-session key usage

Skip until triggered. Do not pre-emptively implement.
```

## v1.2 backlog (out of scope here)

- Long-poll consumer path removal (deprecated in K)
- json CLI transport removal (deprecated in N)
- Per-instance HMAC keypairs default (if O ran)
- Browser dashboard auth (if remote-access signal arrives)
- **Session selector (CLI + UI)**: end-user discoverability of which SM
  session/job is being monitored. Pain point: dashboard surfaces
  governance state but doesn't expose a session picker, and there is
  no `sm` CLI for listing/tailing sessions. Scope:
  - CLI: `sm sessions list` (rows from gov.db `sessions` table + last
    message ts + active flag from `EngineRegistry.refresh_status`),
    `sm sessions tail <id>` (stream that session's bus envelopes).
    New module `tools/sm_cli.py`; wraps `MessageBus` queries.
  - Dashboard: session picker dropdown in header, sourced from
    `/api/registry/active` (already shipped in Task M). Filters all
    panes (decisions, HITL queue, latency, command stream) by
    selected `session_id`. `/events` SSE already carries `session_id`
    per envelope, so the filter is client-side.
  - Default selection: most recently active session, with a "show
    all" pseudo-id for the current behavior.
  - DOD: `sm sessions list` exits 0 with at least the active session
    listed; dashboard header picker switches all panes; existing tests
    stay green; new `tests/test_sm_cli.py` covers list/tail.
  - Estimate: ~1 session.
- **Soak cassette + replay tiers**: cut quota cost of routine soaks
  and remove upstream rate-limit variance from latency numbers. Pain
  point: 30-min ship-gate soak burns 60 real `claude -p` calls; CI
  cannot run it; rate-limit jitter corrupts p95.
  Three tiers:
  - **replay** (free, every CI run / dev) — `--cli-replay
    <cassette.jsonl>` flag on `tools/soak_driver.py`. Replay sleeps
    `recorded_latency_ms` per envelope then emits canned JSON. Tests
    pool/bus/governance plumbing only.
  - **record-cassette** (weekly refresh, Haiku) — new
    `tools/cassette_record.py` runs a real soak with
    `claude -p --model claude-haiku-4-5-20251001` and writes
    `tests/fixtures/soak_cassette_<date>.jsonl`. Cheap baseline
    refresh; catches model-side envelope drift.
  - **ship-gate soak** (minor version cut, default model) — current
    `tools/soak_driver.py --cli-pool-size 2` path. Source of truth
    for ADR-5 absolute latency numbers.
  ADR-17 documents the three-tier model and warns that cassette p95
  is a *relative* regression signal, not an absolute target. ADR-5
  gets a cross-reference: only ship-gate soak feeds the budget.
  - DOD: replay-mode soak runs end-to-end without a `claude` binary
    on PATH; cassette recorder writes a re-runnable artifact;
    `tests/test_soak_replay.py` exercises the replay path; ADR-17
    merged.
  - Estimate: ~1 session.
- **Claude Code lifecycle bridge**: surface Claude Code background
  jobs (`BG <id>`) and `Agent(...)` subagent spawns into SM's bus so
  the dashboard can show them. Pain point: SM today only sees
  conversation/decision events; Claude Code's internal process
  management (background bash jobs, spawned agents) is invisible.
  When the user runs a background soak or spawns a subagent for a
  task, the SM dashboard shows nothing — the operator can't tell
  which job is being governed.
  Scope:
  - Hook surface in Claude Code: needs `BackgroundJobStart` /
    `BackgroundJobEnd` and `AgentSpawn` / `AgentComplete` events.
    If the existing UserPromptSubmit / TaskOutput hooks don't carry
    these, file an upstream feature request OR poll Claude Code's
    task output directory as a shim (`%LOCALAPPDATA%\claude\...\
    tasks\<id>.output`).
  - New module `src/stream_manager/lifecycle_bridge.py`: subscribes
    to the hook stream and emits typed envelopes into MessageBus
    (`event_type=bg_job_start|bg_job_end|agent_spawn|agent_done`).
  - Governance opt-in: bg jobs and agents do NOT auto-route to L4
    alignment; default ALLOW with a `track_only` decision unless
    operator wires a policy. Avoids quota burn from the bridge
    itself.
  - Dashboard panel: "Active jobs / agents" pane listing live
    BG job IDs + agent IDs + start ts + status. Pairs with the v1.2
    session selector — selecting a session filters the panel.
  - DOD: spawning a background bash via Claude Code or an
    `Agent(...)` subagent shows up in the dashboard within 2s;
    completion event closes the row; no governance decision is
    spent on track_only events; `tests/test_lifecycle_bridge.py`
    covers the round-trip.
  - Estimate: ~1–2 sessions (plus 0–1 session of upstream work if
    Claude Code hook surface needs an extension).
