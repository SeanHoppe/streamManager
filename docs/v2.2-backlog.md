## Seed items

### 🟡 Remote-CLI monitoring — extend SM beyond local JSONL tail

**Surfaced 2026-05-08** during issue #128 (PPP probe transport) review.
SM today reaches CLI instances exclusively via local-disk JSONL tail
(`src/stream_manager/jsonl_tail.py`) + local SQLite WAL bus
(`src/stream_manager/message_bus.py`). Bus = local file path. JSONL paths =
local-disk. No HTTP / network surface for SM monitoring.

PPP audit harness (v2.1 P1–P3) audits the SM↔stream binding for streams
SM already reaches; it does NOT extend reach. Question raised at #128
review: can SM monitor remote CLI instances? Answer: not today, no
matter which probe transport is chosen. Out of v2.1 scope. Track here.

**Three implementation sketches (decision belongs to whichever cycle
picks this up):**

- **Option A — JSONL sync.** Remote box rsyncs / Syncthing / SMB mounts
  its `~/.claude/projects/` to a path SM tails locally. Cheapest. Latency
  bounded by sync cadence. Multi-host = N sync paths under one local SM.
- **Option B — Remote bus bridge.** Remote box runs lightweight publisher
  that ships its bus envelopes over SSE / WebSocket to local SM. Re-uses
  ADR-14 SSE shape. Authentication + back-pressure design needed.
- **Option C — Distributed SM agents + roll-up.** Per-host SM instance,
  central dashboard aggregates bus state. Heaviest. Operationally most
  flexible but introduces multi-process governance state coherence
  problems (which agent is authoritative on cross-host patterns?).

**Promotion criterion:** only promote to a v2.x P1 if a concrete
remote-monitoring use case is on deck (e.g. CI runner observation,
multi-developer SM-as-service, remote pair-programming). Speculative
demand alone does not promote.

**Cross-cutting risks (sketch — not load-bearing til promotion):**
- Network surface = new attack surface. ADR review required.
- HMAC seam currently single-process; remote bridge multiplies the
  secret-distribution problem.
- `provenance_assertions` (v2.1 P1) is local SQLite; remote write
  semantics undefined.
- `feedback_no_self_monitor.md` filter widens — SM-watching-SM-on-other-host
  is a new failure mode.

**Refs:**
- Issue #128 (probe transport decision; recorded transport ≠ monitoring transport).
- v2.1 P1 prompt: `docs/prompts/v2.1-orchestration/phase-1-ppp-stream-disambiguation.md`.
- ADR-14 (SSE transport shape — re-usable for Option B).
- `src/stream_manager/jsonl_tail.py`, `src/stream_manager/message_bus.py`.
- Memory `feedback_no_self_monitor.md`.

## ADR-18 Rule 5 — backlog hard cap

This file holds **6 post-graduation seeds** at v2.2 P0 (1
pre-existing remote-CLI + 5 INTENT.md gap-analysis seeds) **+ 5
carry-forwards from v2.2 ship-gate** (items 1–5 above) = **11 open
items going into v2.3 P0**. ADR-18 Rule 5 text caps **per-item**
carry-forward at two cycles without graduation; it does NOT
explicitly bound aggregate backlog size. The v2.0 P4 + v2.1 P4
close-outs both expanded the open-seed count at ship-gate via the
"carry-forwards from v(n)" section without a separate Rule 5
amendment — informal precedent, not codified policy. v2.3 P0 cycle
frame may collapse / fold items per its own pre-flight + scope
decisions; if v2.3 grows the count further the operator should
either explicitly accept another cap-bump (as at v2.2 P0) or mint
a Rule 5 clarifying amendment to ADR-18 documenting the cycle-
handoff exemption pattern verbatim.

**Operator accepted cap-bump 1 → 6 at P0.** ADR-18 Rule 5 caps
further per-cycle growth; v2.2 cycle work that opens any NEW item
beyond this cap must close or graduate ≥ 1 existing seed before
merging.
