## TASK B — Per-session engine instancing refactor

```
Refactor GovernanceEngine from singleton-per-process to one instance per
session_id. This is the foundation for v1.0 multi-session support
(Q3 lock in memory/project_sync_comms.md).

Today: one GovernanceEngine attaches to one MessageBus and processes all
sessions through shared in-process state (DecisionGraph, mode ladder,
pattern counters).

Target: an EngineRegistry holds {session_id: GovernanceEngine}, lazily
constructs an engine per new session_id seen on the bus, and routes
each evaluate() call to the correct instance.

Scope:

1. `src/stream_manager/governance.py`
   - Add `class EngineRegistry`:
     - `__init__(self, bus: MessageBus, project_root: Path, ...)`
     - `get_or_create(session_id: str) -> GovernanceEngine`
     - `close(session_id: str)` — releases an engine when session ends
     - `active_session_ids() -> list[str]`
   - GovernanceEngine __init__ now takes session_id explicitly.
   - All in-memory state (DecisionGraph, mode ladder, pattern counters,
     hitl_state) becomes per-instance, not module-level globals.

2. `src/stream_manager/hook_entry.py` (or wherever PreToolUse fires the hook)
   - Read `session_id` from hook payload (Claude Code provides it)
   - Call `registry.get_or_create(session_id).evaluate(msg)` instead of
     a singleton engine

3. `dashboard/server.py`
   - Server holds `EngineRegistry`, not single engine
   - Existing `/api/sessions` already lists sessions; verify session
     selector still works
   - Add `GET /api/registry/active` returning active session_ids list
     (for debugging / dashboard display)

4. SM-never-self-monitor enforcement:
   - EngineRegistry.get_or_create rejects (raises) if session_id matches
     `os.environ.get("SM_OWN_SESSION_ID")` — set at SM boot.

5. Tests:
   - `tests/test_engine_registry.py` (new):
     - two sessions get distinct DecisionGraphs
     - L4 promotion in session A does NOT flip session B's mode
     - registry.close releases instance
     - SM_OWN_SESSION_ID rejection
   - Update existing `tests/test_governance.py` to use registry where
     appropriate; keep singleton-style tests by constructing engine with
     a specific session_id.

DOD:
- All existing tests pass
- New registry tests pass
- 30-min soak baseline still PASS (run a quick 5-min soak to confirm
  no regression: `python tools/soak_driver.py --total-seconds 300`)
- No shared mutable state across sessions (verify by greppping for
  module-level state in governance.py)
```

## TASK D — `desktop_command` emit + HMAC + WAL

```
Implement the SM → governed-session outbound control plane.
Reference: docs/sync-comms-qa.md Q2 + OQ1–OQ4 locks.
memory/project_sync_comms.md is the canonical design freeze.

Depends on Task B.

Scope:

1. New WAL table in `src/stream_manager/message_bus.py` _SCHEMA:
   CREATE TABLE IF NOT EXISTS desktop_commands (
       id TEXT PRIMARY KEY,
       session_id TEXT NOT NULL,
       kind TEXT NOT NULL,
       args_json TEXT NOT NULL DEFAULT '{}',
       signature TEXT NOT NULL,
       sent_at REAL NOT NULL,
       acked_at REAL,
       status TEXT NOT NULL DEFAULT 'pending',
       error TEXT
   );
   CREATE INDEX IF NOT EXISTS idx_dc_pending ON desktop_commands(session_id, status);

2. New module `src/stream_manager/desktop_commands.py`:
   - SECRET_PATH = ROOT / ".bridge/secret"
   - `_load_or_gen_secret() -> bytes`:
     - If env SM_DESKTOP_SECRET set → use it (utf-8 encode)
     - Else if SECRET_PATH exists → read it
     - Else → secrets.token_urlsafe(32), write to SECRET_PATH with
       os.chmod(path, 0o600), return it
   - `sign(payload: dict) -> str`:
     - Canonical JSON of payload (sorted keys, no whitespace)
     - HMAC-SHA256 hex digest
   - `validate(payload: dict, signature: str) -> bool`:
     - Constant-time compare (hmac.compare_digest)
   - `KIND_ALLOWLIST = {"pause","foreground","flash","audible_cue","surface_hitl","request_attention"}`
   - `emit_command(bus, session_id, kind, args) -> str`:
     - Reject if kind not in allowlist (raise ValueError)
     - Generate uuid4 id, ts, sign payload
     - Insert into desktop_commands table with status='pending'
     - Return id

3. `dashboard/server.py`:
   - `POST /api/commands` body: {session_id, kind, args}
     → calls emit_command, returns {id, status:"pending"}
   - `GET /api/commands/pending?session_id=X`
     → returns list of pending commands with sig + payload
     (TTL: any pending older than 30s gets status='expired' on read,
     not returned)
   - `POST /api/commands/{id}/ack` body: {status: "ok"|"rejected", error?}
     → updates row, sets acked_at
   - All endpoints validate session_id != SM_OWN_SESSION_ID

4. Tests `tests/test_desktop_commands.py` (new):
   - secret auto-gen and 0600 perm (skip 0600 check on Windows)
   - env override beats file
   - sign+validate round-trip
   - kind allowlist rejection
   - TTL expiry on read
   - SM_OWN_SESSION_ID rejection

DOD:
- pytest green (incl. new test file)
- `.bridge/secret` created on first run, mode 0600 (or just exists on Win)
- Manual: POST /api/commands → row in DB with valid sig → GET pending
  returns it → POST ack → status updates
- No engine.evaluate() invoked for any command (validate via log/spy)
```

## TASK E — Governed-side hook handler + ack

```
Wire the consumer side of desktop_command. The governed session's
PreToolUse hook (or a sidecar daemon spawned alongside it) long-polls
SM for pending commands, validates HMAC, executes, posts ack.

Reference: docs/sync-comms-qa.md OQ4 lock — long-poll transport.
Depends on Task D.

Scope:

1. New module `src/stream_manager/desktop_command_consumer.py`:
   - Class `CommandConsumer`:
     - __init__(self, sm_url: str, session_id: str, secret: bytes,
                executors: dict[str, Callable])
     - run_forever(self): loop
       - GET {sm_url}/api/commands/pending?session_id={session_id}
       - For each command:
         - validate signature with shared secret
         - if invalid → POST ack with status='rejected', error='bad_sig'
         - if kind not in self.executors → ack rejected, error='unknown_kind'
         - else: call executor(args), ack with status='ok' (or rejected
           on raise, with error=str(exc)[:200])
       - sleep 1.0s
   - Default executors stub:
     - pause → no-op (placeholder, real impl will hook into Claude Code)
     - foreground/flash/audible_cue → no-op stubs (log only)
     - surface_hitl → already wired via existing HITL plumbing
     - request_attention → log only
   - Single executor signature: `def execute(args: dict) -> None:` (raises on failure)

2. CLI entry point `tools/sm_consumer.py`:
   - argparse: --sm-url, --session-id, --secret-env (default SM_DESKTOP_SECRET)
   - constructs consumer + run_forever
   - This is the daemon a governed session would spawn alongside its
     Claude Code instance.

3. Tests `tests/test_desktop_command_consumer.py` (new):
   - mock SM endpoints; verify:
     - valid command → executor called → ack=ok
     - bad sig → executor NOT called → ack=rejected
     - unknown kind → ack=rejected
     - executor raises → ack=rejected with error text
     - poll loop sleeps 1s between iterations (mock time)

DOD:
- pytest green
- Manual: start SM dashboard, start consumer, POST a command via SM
  /api/commands, observe ack=ok within ~1s on consumer side
- Bad-secret consumer rejects commands (manual test with mismatched env)
```

## TASK F — Cross-session pattern learning

```
Add opt-in cross-session pattern hydration with HITL-gated promotion.
Reference: docs/sync-comms-qa.md Q3 + OQ5/OQ6/OQ8 locks.

Depends on Task B (per-session engines).

Scope:

1. `src/stream_manager/message_bus.py`:
   - Add column to `patterns` table via additive migration:
     `cross_session INTEGER NOT NULL DEFAULT 0`
     (follow the Phase 4 ALTER TABLE pattern at message_bus.py:182)
   - New methods:
     - `flag_pattern_cross_session(hash: str)` — set =1
     - `unflag_pattern_cross_session(hash: str)` — set =0
     - `get_cross_session_patterns() -> list[Pattern]` — flag=1 rows

2. `src/stream_manager/governance.py` (per-session engine):
   - On L3 promotion (existing code path), instead of auto-flagging,
     emit a HITL queue entry: bus.queue_hitl(message_id, action="flag_cross_session",
     trigger_reason="cross_session_flag", proposed_action="flag",
     proposed_confidence=0.9)
   - Operator HITL approval calls bus.flag_pattern_cross_session(hash)
   - Wire HITL resolution dispatcher (in hitl.py) to handle
     resolution=="approved" + trigger_reason="cross_session_flag" → flag

3. New module `src/stream_manager/cross_session_hydrator.py`:
   - Class `Hydrator(threading.Thread)`:
     - __init__(self, engine: GovernanceEngine, bus: MessageBus)
     - run(): bus.get_cross_session_patterns() → for each, inject into
       engine.graph at L1 advisory level (existing graph.add_pattern
       with level=1)
     - Set engine.hydrated = True on completion
   - Engine __init__ spawns Hydrator (daemon=True), engine.hydrated = False
     until thread completes
   - evaluate() does NOT block on hydration; pre-hydration decisions
     simply skip cross-session advisory (graph lookup returns None as
     today for un-injected patterns)

4. Periodic refresh:
   - EngineRegistry adds a 60s timer that calls hydrator-style refresh
     for ALL active engines, picking up new cross-session flags
   - Use `threading.Timer` chained or `asyncio.create_task` if FastAPI loop

5. `dashboard/static/index.html` + `dashboard/server.py`:
   - New panel "Cross-session patterns" listing all flag=1 rows:
     hash, level, occurrences, success_rate, "Demote" button
   - `POST /api/patterns/{hash}/demote` calls bus.unflag_pattern_cross_session
   - No demote on per-decision HITL override (per OQ6)

6. Tests `tests/test_cross_session_learning.py` (new):
   - L3 promotion in session A enters HITL queue (not auto-flagged)
   - Operator approve → flag set
   - Hydrator injects flag=1 patterns into new engine at L1
   - New engine cannot promote inherited L1 to L2 without local re-validation
     (i.e. local occurrence count starts at 0, gates apply)
   - Demote endpoint flips flag back to 0
   - Existing TTL gc still removes stale patterns

DOD:
- pytest green
- patterns.cross_session column visible in `.schema` output
- HITL queue receives entries with trigger_reason=cross_session_flag
- Demote panel works in all 3 themes
- 30-min soak still PASS post-merge
```
