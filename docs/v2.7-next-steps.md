## Cycle-type fallback (consolidation override)

If operator picks **consolidation** at P0 fire (override of any
feature lean that surfaces during fire-time deliberation), the
following dispositions apply:

- All work-phase rows in the §"Fire-order" table become **DEFERRED
  v2.8**.
- Seed v2.6-G step (2) timeout-tighten DEFERS v2.8 (1st defer since
  v2.5 P0 promote-confirmation; step (2) measured eval p99 evidence
  remains in hand and the J2 audit recommendation re-applies at
  v2.8 P0 unchanged).
- Seed v2.6-G step (3) env-split DEFERS v2.8 in lockstep.
- Seed v2.6-C 5th-consecutive deferral; renames Seed v2.7-C at v2.7
  P2 backlog mint.
- Seed v2.6-A n=12 re-measure may still fire at v2.7 P2 under the
  current cap (no `src/` touch required for the re-measure protocol
  itself; J3 protocol fires regardless of cap state).
- Cycle becomes 2-numbered-phase (P0 + P2 ship-gate), 0 work phases.
- Lever ledger HOLD at 2 production / 0 soak (unchanged).
- Operator records the override rationale in `docs/v2.7-task-plan.md`
  §"Operator decisions" #1 (typical: scope uncertainty on cap value,
  dependency on v10.3 stochastic propensities not yet available,
  lever-budget bound, alternation-hygiene preference after v2.6
  feature cycle, etc.).

This fallback exists because alternation hygiene is NOT law; ADR-18
Amendment A §"Default lean rationale" permits two-cycle-in-a-row of
the same type when the candidate-lever backlog has no ready entry.
Conversely, two-consecutive-feature is permitted when ready FIRE
candidates exist (Seed v2.6-G step (2) is ready; Seed v2.6-C is
ready iff v10.3 dependencies are accounted for). Operator judges at
P0 fire time.
