### 🟡 Haiku fastpath confidence floor — lever fires 0% under production load

**Added:** v1.8 P2 ship-gate close-out (2026-05-06)

**Why:** v1.8 P1 wired `is_ambiguous_block` / `is_hitl_synthesis` at the pre-routing call site. Unit tests confirm routing is correct. Ship-gate soak (60 events, 32.1 min, `reports/soak-20260506T101746Z.md`) recorded `cli_dispatch_fallback_ms` p95 = 0.00 and 0 `governance_fallback_routed` envelopes. Root cause: Haiku returns confidence ≥ 0.70 on all tested ambiguous-block content (short imperative destructive commands, deliberative question forms). The `BRIDGE_L4_FALLBACK_CONFIDENCE` floor (0.70 default) is never crossed. The Haiku-first path IS being taken (routing is wired), but the Sonnet retry never fires.

Two P2a attempts (deliberative questions → P1c-B; imperative declarative forms → P2a) both produced 0 fire rate. Third attempt not warranted without a mechanism change.

**Options for v1.9:**
1. Lower `BRIDGE_L4_FALLBACK_CONFIDENCE` from 0.70 to ~0.50 — risks always-fallback if Haiku returns 0.50-0.70 uniformly on borderline content (must measure first).
2. Restrict `is_ambiguous_block` to HITL-synthesis-only path — the HITL trigger is more deterministic (based on session state, not content heuristic); more reliable lever activation.
3. Replace confidence-based fallback trigger with verdict-based: if Haiku returns BLOCK on ambiguous-block content, auto-retry with Sonnet for a second opinion (more predictable trigger than confidence floor).
4. Expand the `is_ambiguous_block` heuristic to broader content patterns that Haiku genuinely finds borderline (requires empirical study of Haiku confidence distribution on governance prompts).

**Scope:** `src/stream_manager/cli_governance.py` (floor constant), `src/stream_manager/governance.py` (heuristic), `tools/soak_driver.py` (corpus). Alignment-eval `--ci-gate` re-run required.

**References:** `docs/adr/ADR-5-latency-budget.md` §"v1.8 ship-gate baseline" §"Caveats", `reports/soak-20260506T101746Z.md`, v1.8 P2a `docs/prompts/v1.8-orchestration/phase-2a-soak-prompt-coverage.md`.

### 🟡 External session registry — SM discovers and tracks non-SM `claude -p` subprocesses

**Added:** UC-01 lessons-learned capture (2026-05-06)

**Why:** On 2026-05-06 an operator spent 7 manual steps to answer "is the certPortal Matt QA agent healthy?". SM had zero visibility into the certPortal oversight pipeline running as isolated `claude -p` subprocesses. Session state was discoverable (via `~/.claude/sessions/*.json` + `ps`), but SM had no mechanism to ingest it. Full incident in `docs/use-cases/uc-01-external-session-monitoring.md`.

**How to apply:** Poll or inotify-watch `~/.claude/sessions/` for JSON files. On change, read pid/sessionId/cwd/entrypoint; cross-check liveness with `kill -0`. Emit `external_session_registered` / `external_session_exited` bus envelopes. Dashboard surface: active external sessions panel with liveness indicator and last-seen time.

**Scope:** new `src/stream_manager/session_watcher.py`; bus envelope schema; dashboard panel. No changes to governance engine — read-only observation path.

**Acceptance criteria:** `docs/use-cases/uc-01-external-session-monitoring.md` AC-1.

### 🟡 Learn Mode JSONL expansion — ingest configurable external session sources

**Added:** UC-01 lessons-learned capture (2026-05-06)

**Why:** Learn Mode (`v1.3`) ingests only the Desktop session JSONL. certPortal oversight agents (Dave/Jen/Jason/Matt/Oliver/Michael) produce rich, high-quality QA dialogue across hundreds of turns per day. This is exactly the signal SM's categoriser should learn from — PASS/FAIL reasoning, retry patterns, escalation triggers — but the ingest pipeline never sees it.

**How to apply:** Add a `learn_sources` config list (cwd globs or explicit JSONL paths). Each source is a named label (`certportal-oversight`). Ingest pipeline tags turns with source label before categorisation so oversight-agent patterns are attributed separately from SM-governance patterns. Gate-keeper: if source label matches an SM-internal session, reject (per `feedback_no_self_monitor.md` — no eval feedback loops). Default off; opt-in per source.

**Scope:** `src/stream_manager/learn_mode.py` (source config + label tagging); `docs/learn-mode-design.md` update; tests for source isolation.

**Acceptance criteria:** `docs/use-cases/uc-01-external-session-monitoring.md` AC-2.

### 🟢 Background task token registry — bus event + dashboard visibility for bg tokens

**Added:** UC-01 lessons-learned capture (2026-05-06)

**Why:** `b1bu59d1l`-style background Bash task tokens are opaque to SM. The operator had to know that: (a) task output files are 0 bytes during execution, (b) live output goes to a tee log, and (c) the file path encodes the originating sessionId. Dashboard visibility would surface token age, output file path, and size-change event without any manual archaeology.

**How to apply:** When the session watcher (item 4 above) detects a JSONL entry with `backgroundTaskId`, record: token, output file path, originating session, start time. Poll output file size every 30s; emit `bg_task_output_ready` bus envelope when size transitions from 0. Dashboard: pending bg tokens panel with age, path, size. Operator can trigger a Monitor on any token from UI (fires the pattern in UC-01 §"Monitor pattern").

**Scope:** `session_watcher.py` (token extraction from JSONL); bus envelope; dashboard panel. Depends on external session registry (item 4) being implemented first.

**Acceptance criteria:** `docs/use-cases/uc-01-external-session-monitoring.md` AC-3.
