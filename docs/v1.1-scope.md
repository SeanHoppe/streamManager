# StreamManager v1.1 Scope

**Status:** draft, post-v1.0.0
**Audience:** maintainer + future Claude Code session running task plan
**Companion:** `docs/v1.1-task-plan.md` (concrete task block series)

---

## Goals

v1.0 shipped the sync-comms scope freeze: per-session engines, HMAC desktop_command, cross-session HITL learning, Session Mirror. Two ship-gate gaps surfaced during validation:

1. **p95 latency regression** — 19.08s vs ADR-5 budget 15s. Likely Hydrator + cross-session DB read on engine construction. Top priority.
2. **OQ4 long-poll** delivers commands at ~1s steady-state. Push toward sub-second.

Plus a small backlog of items deferred from v1.0 spec freeze:
- CLI subprocess warm-pool (latency)
- WireCLI re-incorporation
- A few schema / wiring hygiene items surfaced during v1.0 implementation

v1.1 is **performance + hygiene**, not new features. No new FR-DC / NFR-MS additions. ADR-13 (single-secret HMAC) stays; per-instance keypairs deferred unless a multi-tenant signal arrives.

## Latency budget targets (revised)

ADR-5 v1.0:
- p50 ≤ 7s
- p95 ≤ 15s
- hard timeout 25s

ADR-5 v1.1 target (post warm-pool + Hydrator fix):
- p50 ≤ 3s
- p95 ≤ 8s
- hard timeout 25s (unchanged)

If P0+J ships and measured numbers don't meet the v1.1 target, re-baseline the budget rather than blocking the release. Performance is iterative — set the next floor, don't overpromise.

---

## Risk register

- **Hydrator profiling could reveal it's NOT the p95 cause.** Real cause might be CLI subprocess cold-start variance under host load. Task I starts with measurement before refactor.
- **Warm-pool process management on Windows** is fragile. The `claude` CLI subprocess doesn't stream-multiplex; pool is N idle processes. Risk: orphaned children on dashboard crash. Mitigation: PID file + SIGTERM-on-shutdown handler, `psutil.process_iter` reaper at boot.
- **Long-poll → SSE migration** breaks the existing consumer protocol. Need versioned endpoint (`/api/commands/stream` alongside `/api/commands/pending`) so old consumers keep working until cut over.
- **Schema migration for cross_session HITL** (Task L) requires careful additive ALTER. Existing rows have hash encoded in `proposed_action="flag_cross_session:{hash}"` — need backfill on migrate.
