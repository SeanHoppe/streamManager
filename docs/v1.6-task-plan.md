## PHASE P1 ŌĆö `_evaluate_inner` CLI residue instrumentation

**Pain point:** v1.5 ship-gate (`reports/soak-20260504T201714Z.md`) FALSIFIED the v1.4 sub-phase-tail hypothesis. ALLOW p95 = 5.60 s, of which the five v1.5 sub-phases sum to **0.13 ms p95** vs `evaluate_inner` p95 = **5599 ms**. Without residue-level attribution, ADR-5 cannot localize the tail to a specific component (model wall-clock vs pool acquisition vs CLI parse) and cannot defensibly choose the v1.7 lever (Haiku fastpath vs pool sizing vs other).

**References:**
- `docs/adr/ADR-5-latency-budget.md` v1.5 ┬¦"Caveats"

- `tools/soak_driver.py` ŌĆö `_ALLOW_PHASE_ORDER` (extend), `_format_allow_phase_breakout`

**Scope:**

1. Add five new sub-phase wall-clock keys on `engine._last_phase_timings_ms` for ALLOW envelopes that traverse the CLI escalation branch:

   | Sub-phase key | Spans |
   |---|---|
   | `cli_setup_ms` | entry of `_maybe_cli_evaluate` ŌåÆ just before `CliGovernor.evaluate(...)` invocation (covers lazy `CliGovernor()` construction + `_system_prompt()` cache build) |
   | `cli_dispatch_ms` | full duration of `CliGovernor.evaluate(...)` from caller's perspective (entry ŌåÆ return) |
   | `cli_pool_acquire_ms` | duration of `with self._pool.acquire() as worker:` enter ŌåÆ exit (pool-path only) |
   | `cli_pool_send_ms` | duration of `worker.send(pool_prompt, timeout=...)` (pool-path only). Spawn-path: covers `subprocess.run(...)` round-trip. |
   | `cli_parse_ms` | duration of `_extract_usage(stdout)` + `_parse_envelope(stdout)` (both paths) |

   Branches that do NOT call the CLI (precheck-hit, high-confidence graph match, FR-OG-7 override, FR-AR-6 blocked-op short-circuit) MUST emit each of the five new keys with value `0.0` so soak rows are dense and percentile math is honest about the precheck-hit ratio.

When non-None, populate `cli_dispatch_ms`, `cli_pool_acquire_ms`, `cli_pool_send_ms`, `cli_parse_ms` into the dict from the appropriate timing capture sites. When `None`, behavior is identical to v1.5 (back-compat for existing callers/tests that build `CliGovernor` directly).

3. Caller in `governance._maybe_cli_evaluate` reads `self._sub_timings_in_flight` (the v1.5 transient slot) and threads that dict through to `CliGovernor.evaluate(content, model_id=chosen, sub_timings=<dict>)`. Capture `cli_setup_ms` in `_maybe_cli_evaluate` itself before the call.

   - Render a third markdown block `### ALLOW _evaluate_inner CLI residue breakout (v1.6)` from the same `engine._last_phase_timings_ms` source. Same n / p50 / p95 / max columns. Header text references ADR-5 v1.5 ┬¦"Caveats" as the diagnosing target.
   - Place it directly after the v1.5 sub-phase block in the report.
   - Suppress for pre-v1.6 streams (any record missing all five new keys).

5. Two new tests:
   - `tests/test_governance_cli_residue_timings.py` ŌĆö assert all five new keys are populated on the CLI escalation branch (precheck=None, graph=None); assert all five are `0.0` on a precheck-hit branch; assert verdict equality vs a baseline run; assert `cli_dispatch_ms` Ōēź `cli_pool_send_ms` (parent Ōēź child) on the pool-path; assert v1.4 + v1.5 keys still present.
