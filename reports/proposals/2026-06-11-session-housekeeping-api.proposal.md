# Session housekeeping API: stale-session discovery and bulk-purge endpoints with cascade cleanup

**Status:** Proposal (EXPERIMENTAL spike) -- NOT authorized to edit FROZEN surfaces.
**Source:** sm-ux-propose (flywheel) 2026-06-11; idea BACKEND-1; boldness SAFE; refute verdict CONSTRAIN; effort M.
**Scope:** a written proposal only (Rosetta /report-fixes proposal-half). Shipping requires a normal v2.x cycle frame + the noted ADR amendments.

## Problem

Operators must manually poll /api/sessions or scan gov.db with grep to find sessions that ended >24h ago. No automated stale-session visibility or purge surface exists; cleanup is ad-hoc, requiring direct database access or curl commands to delete cascade rows. Multi-week soak runs accumulate sessions (~200+ rows per cycle), inflating SQLite WAL compaction costs and operator friction.

## Proposal

Add two new REST endpoints to dashboard/server.py: GET /api/sessions/stale?retention_hours=24&include_metrics=true (returns time-sorted array of {id, project_slug, ended_at, last_message_ts, decision_count, pattern_count, can_purge_reason}), and POST /api/sessions/bulk-cleanup (accepts {session_ids: [list]}, purges cascade rows, fires session_ended SSE event). Implement new MessageBus.purge_session(session_id) method executing cascade-delete on messages, decisions, hitl_pending, agents, desktop_commands, provenance_assertions in a single transaction, then soft-delete session row (set cleanup_ts=NOW()). Add session_ended to ENVELOPE_KINDS in src/stream_manager/envelope_kinds.py. Fire bus.write_envelope("session_ended", {session_id, ended_at}) when MessageBus.purge_session() is called. Additive schema migration: ALTER TABLE sessions ADD COLUMN cleanup_ts REAL (idempotent via contextlib.suppress). CONSTRAINTS (mandatory per refuter): (1) /api/sessions/stale query MUST filter WHERE project_slug NOT IN ('streamManager') AND session_id != SM_OWN_SESSION_ID to prevent SM self-monitoring (G2 polarity). (2) session_ended is a new envelope kind; same-PR coverage required in tools/cassette_record.py (new _record_session_envelopes function) and tools/soak_driver.py replay logic. (3) cleanup_ts column migration follows lines 495-602 pattern in message_bus.py (contextlib.suppress wrapper for idempotence). (4) Cascade purge implementation: DELETE messages WHERE session_id=?, then DELETE decisions/hitl_pending/agents/desktop_commands/provenance_assertions by session_id FK, under self._lock in a single transaction. (5) POST /api/sessions/bulk-cleanup MUST reject with HTTP 400 if any session_id in the list equals SM_OWN_SESSION_ID before processing (duplicate guard on /api/commands pattern at dashboard line 2407). (6) SSE filter in dashboard sse_events generator (line 2715) must exclude session_ended rows matching SM_OWN_SESSION_ID, matching behavior at line 2740.

## Operator value

Removes manual stale-session discovery and purge friction; enables dashboard self-service cleanup comfort via checkbox-grid UI; defrays database bloat over multi-week soak runs (v10 RL, ship-gate cycles). Surfaces session retention policy as data (retention_hours query param), not hard-coded. MEDIUM value: genuine usability win (reduces operator DB-access friction) but NOT critical path -- operators can still curl sql or query sqlite3 CLI directly; soak driver can purge on shutdown via Python one-liners. Polling anti-pattern (5-minute poll) is acceptable for housekeeping; more elegant = MessageBus.end_session() fires session_ended immediately, but that is out-of-scope design refinement.
