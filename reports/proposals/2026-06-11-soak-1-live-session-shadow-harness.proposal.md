# Live Non-SM Session Soak Dashboard Harness

**Status:** Proposal (EXPERIMENTAL spike) -- NOT authorized to edit FROZEN surfaces.
**Source:** sm-ux-propose (ux) 2026-06-11; idea SOAK-1; boldness STRETCH; refute verdict CONSTRAIN; effort L.
**Scope:** a written proposal only (Rosetta /report-fixes proposal-half). Shipping requires a normal v2.x cycle frame + the noted ADR amendments.

## Problem

The binding MVP gate (#112 blocker) is blocked on empirical shadow validation (non-SM live session soak), but the `soak_driver.py` tiers (Tier 1-3) are all synthetic or cassette-based. There is no UI to deterministically pick a live non-SM target session, initiate a shadow soak, or watch the polarity-guarded tail->bus->governance chain exercise in real time. Operators currently run soak_driver manually, wait 30 minutes in a blind spot, then parse markdown reports by hand to validate polarity gates + coverage gaps.

## Proposal

Promote the live non-SM session soak from ad-hoc hand-run to a first-class dashboard feature. In Frame C (Background Jobs), add a **Soak Control Panel** (new subsection) with: (1) a dropdown `Select session to soak:` populated from `/api/sessions` filtered by `project_slug != streamManager AND session_id != SM_SELF`, ranked by recency + recent-message density (busy = recent envelope count > 0 in last 60s); (2) a `Start Shadow Soak` button that POSTs to `/api/soak/init` with the target session_id, spawning `tools/soak_driver.py --live-session <id> --duration 300 --bind-output /dev/shm/soak-<id>.jsonl`; (3) a live **soak progress panel** showing: current time elapsed, envelope count tailed, decision breakdown (ALLOW/SUGGEST/GUIDE/INTERVENE/BLOCK stacked bar), synthetic-vs-live coverage delta, and streaming tail latencies (median/p95); (4) on completion, the backend emits a `soak_ready_for_ship_gate` internal bus message; the dashboard HITL subsystem converts this into a new `hitl_pending` row with `proposed_action='approve_soak_gate'` and `trigger_reason='soak_completion_pending_review'`; (5) Frame A (HITL foreground) renders the escalation modal: "Ship Gate Ready?" with a comparative report card (synthetic tier benchmark vs live-soak results) highlighting polarity-gate validation, coverage gaps, and a verdict button (APPROVED/DISMISSED); (6) operator resolves via `/api/hitl/resolve`, decision persists; the soak comparative report (read-only narrative) stays in Frame C as observability. The whole feature is gated on HITL mode ON or explicit operator opt-in (safety: prevents accidental long-running soaks). Reports persist to `reports/soak-live-<session-id>-<ISO-ts>.md` using the same shape as Tier 3. This keeps the escalation-only foreground rule intact: all proposed_action controls live in Frame A HITL, not in background panels.

## Operator value

Operator can go from 'is the soak path working?' to 'is this live session valid for the empirical MVP gate?' in one click, with live visual feedback + automated polarity + coverage validation. Closes #112 blocker by providing a gated, reproducible harness for the non-SM-session path. Real-time observability eliminates the 30-minute blind spot; gated decision point (single bounded HITL question) replaces manual report parsing.

## Feasibility

All surfaces are additive to the existing API contract. (1) `/api/sessions` endpoint already exists (dashboard/server.py:720); filter by project_slug + session_id is a WHERE clause addition. (2) `/api/soak/init`, `/api/soak/status/{soak_id}` are new endpoints (~400 LOC for handlers + state tracking). (3) `tools/soak_driver.py --live-session <id>` is a configuration flag extension (~200 LOC, re-uses existing WalReader for live message pump). (4) BackgroundJobsPanel.svelte subsection + SSE consumer (~300 LOC Svelte + polling logic). (5) Report generation re-uses existing soak-report template; no new bus envelope kind required at the soak-driver side. (6) Polarity-gate assertions via SQL query over the soak's local decision log + bus `provenance_assertions` table (no governance.py mutations). (7) The `soak_ready_for_ship_gate` internal message and `hitl_pending` insert are ~100 LOC in governance.py, ~50 LOC in dashboard HITL resolver. No FROZEN table mutations (governance.py, message_bus.py, cli_governance.py, model_router.py stay read-only). Test coverage: existing cassette + soak test paths already exercise HITL routing; no new cassette kind required.

## Compliance (adversarial-refute floor)

- **Firewall (G1):** PASS. The proposal filters by `project_slug != streamManager AND session_id != SM_SELF` at both the `/api/sessions` query and the soak-init POST. This prevents SM from monitoring itself. The four governance surfaces (governance.py, message_bus.py, cli_governance.py, model_router.py) are READ-ONLY; no new certPortal vocabulary is introduced. The new bus message kind (`soak_ready_for_ship_gate`) is internal-direction only (does not trigger SSE fan) and is additive to the existing ADR-14 SSE surface. No new certPortal coupling beyond already-designed (learn-mode source registry, project_context, agent_profiles).
- **Polarity (G2):** PASS. Self-monitoring is architecturally prevented at the endpoint layer + HITL gate. The critical filter `project_slug != streamManager AND session_id != SM_SELF` is enforced on `/api/sessions` (server-side filter at line 735-741 of dashboard/server.py already scopes to non-self) and on soak-init POST. The soak reports persist to `reports/soak-live-<session-id>-<ISO-ts>.md` keyed on the TARGET session, not SM-self. The proposal does NOT introduce any code path that sweeps stale sessions or performs cleanup that could accidentally include SM-self. Operator HITL gate enforces explicit opt-in (safety: prevents accidental long-running soaks). Polarity-gate assertions in the comparative report auto-validate `project_slug filter confirmed` + SM-self exclusion as read-only observability, not as a control.
- **ADR-18 MUST floor:** CONSTRAINED. ADR-18 4.2 MUSTs: (1) 3-frame presence PASS: Frame C background jobs subsection, non-modal. (2) Escalation-only foreground BEND->CONSTRAIN: The original proposal placed the "Ship Gate Ready?" verdict button inline in Frame C soak progress panel, violating the escalation-only rule. CONSTRAINT requires: (a) On soak completion, emit `soak_ready_for_ship_gate` bus message (internal, no SSE fan). (b) Dashboard HITL subsystem converts to new `hitl_pending` row with `proposed_action='approve_soak_gate'` + `trigger_reason='soak_completion_pending_review'`. (c) Frame A renders the verdict as a bounded escalation modal (existing HITL pattern), not as a Frame C control. (d) Operator resolves via `/api/hitl/resolve`; comparative report stays in Frame C as read-only narrative. This preserves M2 escalation-only foreground: all proposed_action controls live in Frame A HITL. (3) Paired label+color badges PASS: decision breakdown uses paired labels + color. Envelope badges similarly labeled. (4) Absolute HITL gate PASS: gated on HITL mode ON or explicit operator opt-in. (5) Domain-agnostic PASS: Session filtering by ID + project_slug; no hard-coded role/job vocabulary. (6) a11y axe gate UNKNOWN (design unspecified; assume compliant pending review). (7) Latency budget PASS: soak driver inherits ADR-5 infrastructure; p50/p95 on live latencies roll into comparative report. (8) Non-goals PASS: no IDE/multiplexer/multi-tenant footprint.
- **Frozen-surface note:** The proposal touches zero FROZEN surfaces per ADR-18 Rule 1. It is purely additive: new endpoints on dashboard/server.py, new Svelte subsection in FrameC_Jobs (composition only, no LifecyclePanel/AsyncHitlQueue changes), new escalation modal mount in FrameA_Sessions (slot-based, no existing pending mount changed). The `soak_ready_for_ship_gate` message is a new internal bus envelope (metadata-only, additive per v10 P4 B' governance_decision pattern). No rip/mutate of governance.py, message_bus.py, cli_governance.py, model_router.py required.
- **New-envelope note:** The proposal introduces one new internal bus envelope: `soak_ready_for_ship_gate` (direction='internal', does not fan to SSE). Schema (FROZEN, metadata-only extensions thereafter): { kind: 'soak_ready_for_ship_gate', soak_id: str, target_session_id: str, timestamp: float }. The envelope is recorded to gov.db via MessageBus.record_internal (does not call fan_envelope, so no SSE broadcast). No new cassette kind required: soak cassettes remain Tier 1-3 shape. HITL routing (conversion to hitl_pending row) is tested via existing HITL cassette paths (dashboard/server.py already has /api/hitl/resolve test fixtures); the soak-specific trigger_reason is a string enum extension, not a new message kind. Soak_driver.py does not emit the envelope; governance.py/dashboard.server.py emits it. Test coverage: extend existing test_hitl.py or test_dashboard_hitl_*.py pattern to cover the `approve_soak_gate` proposed_action insertion and resolution flow. No new cassette_record.py entry needed.
