What v1 ships (Phase 1)

The 2-pane replay-diff modal ONLY. One read against the EXISTING decisions + messages tables -- no new durable table, no governance.py hook, no bus envelope. Additive read endpoint GET /api/decisions/replay-diff, polarity-filtered server-side (mirrors the /api/hitl/pending/triage WHERE-clause).

Path to value: ~2 clicks + 2 handle drags -- open, confirm scope, bracket A, bracket B. No CSV grep, no SQL, no mental reconstruction.

Deferred to Phase 2 (NOT in v1)

Policy-mutation timeline + the policy_snapshots table + a governance.py config-mutation hook. That set touches FROZEN surfaces, so it is a separate future cycle that requires an ADR-18 amendment.

v1 is deliberately valuable WITHOUT policy context: side-by-side proof of behaviour change reads off existing decision history alone.

WARN Build gated on ADR-18 amendment. This UI approach is approvable as drawn, but the upstream brief is flagged NEEDS-AMENDMENT: its prose conflated the Phase-2 policy_snapshots table (a FROZEN-file schema mutation) into v1. The remedy is to ship Phase 1 only (frozenTouch=false, newTable=false, additive read endpoint over existing tables). Code does not start until the corrected v1-only footprint is confirmed; the Phase-2 table + governance hook remain blocked behind a future amendment.
