What this confirms before any code

  • No new frame / no new pane / no new backend. The feature is an additive, gated wrapper at the EXISTING slot=header SessionPicker. OPEN ACTIONS reads the actionCountsBySession map App.svelte already builds from the unscoped /api/hitl/pending poll -- zero new endpoint, zero new field.
  • Three affordances, one scope. GLANCE (chips narrow the option list), CLICK (star pins favorites to the top of picker + rail, persisted in localStorage), KEY (Ctrl+K fuzzy-search over project_slug, Enter switches scope through the guarded selectSession()).
  • Paired label + colour everywhere (M4). Every chip carries a literal count suffix as TEXT; the ACTIVE chip pairs a slate dot + the word ACTIVE + the integer; OPEN ACTIONS pairs the existing amber identity + the word + the integer (chrome, no pulse -- never an M2 escalation); a favorite pairs a filled star glyph + aria-pressed + the word "favorite" in its aria-label, in a desaturated accent deliberately distinct from amber.
  • Polarity floor (G2). The SM-self session is never an option, a favorite, or a search hit -- the store self-excludes and Ctrl+K Enter routes through the guarded setter. The mock carries an ownSessionId only to prove it is dropped.
  • Domain-agnostic (M16). Slugs (aurora-ingest / borealis-api / cobalt-batch / delta-worker) are invented generic names rendered FROM DATA -- never a monitored-project's real slug.
  • Honesty note (in the brief). "ACTIVE" is scoped to ended_at == null (running), because /api/sessions carries no last_msg_ts; a true "active in last 5 min" predicate would need the #32 health-digest endpoint -- out of this footprint.
  • New flag key. Registers as session-quick-filter-presets #36 in the existing session group; the key quick-filters #22 is a DIFFERENT feature (HITL posture presets) -- a new key avoids cross-gating.
