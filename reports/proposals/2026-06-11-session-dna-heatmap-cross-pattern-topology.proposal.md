# Session DNA Heatmap: Cross-session pattern topology visualization with confidence per-session

**Status:** Proposal (EXPERIMENTAL spike) -- NOT authorized to edit FROZEN surfaces.
**Source:** sm-ux-propose (flywheel) 2026-06-11; idea WILDCARD-3; boldness STRETCH; refute verdict CONSTRAIN; effort M.
**Scope:** a written proposal only (Rosetta /report-fixes proposal-half). Shipping requires a normal v2.x cycle frame + the noted ADR amendments.

## Problem

Operators manage multiple concurrent sessions (per FR-OG-8 session picker). When a behavioral pattern fires in one session (e.g., `git push --force` in Dev), the pattern may be dormant or weak in other concurrent sessions, below the operator's attention floor. This asymmetry across the session grid is invisible until a pattern explodes into simultaneous blocks in multiple sessions -- a reactive, late-stage discovery. Per FR-MS-2, cross-session L3 pattern context is stored but never visualized. An operator has no way to see pattern propagation vectors: which patterns are spreading across sessions, which are isolated to a single agent, or which have high confidence in one session but are dormant in another (a 26pp confidence gap suggests a governance asymmetry worth investigating).

## Proposal

Add an optional SessionDnaHeatmap component (collapsible sub-pane in Frame C or a new Frame D overlay toggle) that visualizes cross-session pattern topology as a directed graph (session nodes + pattern-hash edges, rendered as a radial or chord diagram via D3). Each session node is labeled with session_id slug + agent roster (FROM DATA per M16). Each edge represents a shared pattern hash with color intensity = pattern confidence in that session (red = high 0.7+, yellow = medium 0.4-0.7, blue = learning < 0.4), paired with a text label or tooltip showing the numeric confidence value (e.g. "0.78") to satisfy M4 (paired label + color, no color-alone signaling). A click on any session node zooms into a SessionDnaCard detail panel showing: (a) top 5 most-confident patterns unique to that session; (b) top 5 shared patterns across >= 2 sessions (contagion cluster) with per-session confidence values. Mousing over a shared pattern edge highlights all occurrences across the session cluster, showing "Pattern X fires in Dev (0.78) and Reviewer (0.52)". BACKEND COST: new GET /api/patterns/cross-session-topology endpoint that queries patterns + decisions tables, groups by matched_hash + session_id, aggregates mean confidence per (hash, session) pair, and returns graph structure {nodes: [{id, slug, agent_ids, profile_slugs}, ...], edges: [{source_session, target_session, hash, confidence_source, confidence_target}, ...], patterns: {hash: {payload, level}, ...}}. No FROZEN schema changes; query is post-hoc observability only (latency budget <500ms p95 on 50k-decision corpus; implement server-side 5-minute memoization if exceeded). ACCESSIBILITY: heatmap components MUST pass axe audit; all interactive nodes require keyboard navigation (Tab through nodes, Enter/Space to open detail card), ARIA labels on every node/edge, focus ring via focus.css (2px #d97706), aria-live regions for state changes ("Detail card opened for Session Dev showing 5 patterns"). NO BUS ENVELOPE KIND CHANGE -- the API uses patterns + decisions (existing tables); the response is a new read-only shape, not a new governance event kind (no cassette/soak requirement per NEW BUS ENVELOPE RULE, but unit tests in dashboard/test/ covering a 3-session, 2-pattern fixture via contract-fixtures.js pattern required before ship). SELF-EXCLUSION: the SM session is filtered from the node roster the same way other observational panels exclude it (per M15 Session Mirror pattern), so SM never sees itself as a governed target.

## Operator value

Operators spot pattern propagation vectors across concurrent sessions -- which patterns are spreading (high confidence across 2+ sessions), which are isolated (single session only), and which show confidence asymmetry (high in Dev, dormant in Reviewer). Early intervention on emerging cross-session patterns closes feedback loops: if a governance decision in one session (e.g., GUIDE on `git push --force` in Dev) informs a similar weak pattern in another session (e.g., 0.52 confidence in Reviewer), the operator can proactively tighten governance consistency. Helps detect when a learned rule in one session should have informed another. On a laptop monitor with 5-7 concurrent sessions, the heatmap is a single glance that shows which patterns are propagating and which asymmetries exist, reducing the cognitive load of cross-session governance alignment. Value is MEDIUM (not HIGH) because it visualizes existing data already queryable via /api/patterns/cross_session -- the heatmap is a presentation enhancement, not a new capability. But the presentation matters: operators are more likely to act on a visual "Session A (0.78) vs Session B (0.32)" signal than on a tabular list.

## Feasibility

HARD (not infeasible, but non-trivial engineering). The idea is buildable in 2-3 phases: (1) backend endpoint + fixture tests (1-2 days); (2) SessionDnaHeatmap + SessionDnaCard Svelte components with D3 radial/chord rendering (2-3 days, includes responsive sizing, edge-label pairing, mouse interaction); (3) a11y audit + keyboard nav implementation + axe gate (1 day). The main cost drivers: (a) D3 chord/radial diagrams are fragile (path generation, responsive canvas/SVG, zoom constraints); (b) cross-session confidence aggregation requires careful JOINs + GROUP BY; (c) the heatmap is a new visualization component with high accessibility bar (keyboard nav, ARIA, focus management). No new bus semantics, no FROZEN touches, no schema extension required -- all work is additive to EVOLVING surfaces (ui-next/, dashboard/server.py read-only endpoint). Indexing strategy on decisions(matched_hash, session_id) is critical to hitting the <500ms budget; this must be documented in a query-plan comment block at server.py endpoint definition. If the query exceeds budget on the 50k-decision test corpus, implement server-side memoization (5-minute TTL cache, invalidated on new pattern upsert or cross_session flag change) -- no additional complexity, adds ~20 LOC of cache logic.

## Compliance (adversarial-refute floor)

- **Firewall (G1):** PASS. No certPortal coupling. Session nodes (id, slug, agent_id roster, profile_slug) are rendered FROM DATA via the endpoint response. No hard-coded JOB-IDs, roles, or governed-target vocabulary. The heatmap is a generic graph viz that treats sessions as opaque data entities.
- **Polarity (G2):** PASS. The heatmap shows cross-session pattern topology (e.g., 'Pattern X fires in Dev [0.78] and Reviewer [0.52]'). Sessions rendered are non-SM sessions per M15 (the SM session is self-excluded from the node roster, same filter logic used by other observational panels). Frame C integration does not govern SM's own session. The FrameC toggle is a display-only surface, not a monitoring/governance layer over SM itself.
- **ADR-18 MUST floor:** BENDS-SHOULD (not VIOLATES-MUST, with required constraints). ADR-18 M1-M9 + UI-DESIGN-SPEC.md M1-M19 define the floor. The proposal integrates as an optional collapsible sub-pane (not a new mandatory frame), so it BENDS the '3-frame guaranteed presence' rule (Frame C's sub-composition is free; the MUST is '3 frames reachable', not '3 frames exactly'). This is permitted per ADR-20: 'The redesign MAY restructure the frame geometry [but] invariants from INTENT remain MUST'. The three activity domains (REPL, Sub-Agents, Jobs) remain reachable; the heatmap is a view toggle, not a replacement. CONSTRAINTS TO COMPLY: (1) M4 (paired label + color): color intensity ALONE is not a signal. Every color swatch (red/yellow/blue) MUST be paired with text label showing confidence value (e.g., '0.78') or band (high/medium/low). In chord diagram: edge label OR tooltip on hover OR detail card with color + numeric confidence. ASCII-only (cp1252-safe) -- use '--' not em-dash. (2) M17 (a11y axe gate): MUST pass axe audit. Requirements: Tab through all nodes, Enter/Space to open detail card, ARIA labels on every node/edge/button, focus ring (2px #d97706), aria-live regions announcing state, Escape dismisses card. (3) M18 (latency ADR-5): endpoint MUST complete <500ms p95 on 50k-decision corpus. If exceeded, implement server-side 5-minute memoization with invalidation on pattern upsert/cross_session flag change. Document query plan + index strategy in server.py. (4) M16 (domain-agnostic): render agent roster as data slugs FROM agents table (first 8 chars UUID + profile_slug if available). No hard-coded roles. (5) Unit tests: dashboard/test/ contract-fixture covering 3-session 2-pattern corpus + confidence aggregation + edge cases required before ship.
