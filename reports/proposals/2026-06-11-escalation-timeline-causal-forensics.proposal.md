# Escalation Timeline: Forensic causal-chain visibility for governance decisions

**Status:** Proposal (EXPERIMENTAL spike) -- NOT authorized to edit FROZEN surfaces.
**Source:** sm-ux-propose (flywheel) 2026-06-11; idea MONITOR-2; boldness STRETCH; refute verdict CONSTRAIN; effort M.
**Scope:** a written proposal only (Rosetta /report-fixes proposal-half). Shipping requires a normal v2.x cycle frame + the noted ADR amendments.

## Problem

When a session escalates (e.g., desktop_pause fires, negative_regression blocks a message), the operator sees the result but not the *causal chain*. 'Why did this message trigger BLOCKED?' requires drilling into the decision details, the previous 5 decisions, and cross-referencing with agent activity during the ~10s window around the event. The decision history is latent in the bus; visibility is zero. Without context, an escalation feels punitive rather than corrective.

## Proposal

Introduce a collapsible 'Escalation Timeline' pane (right-side column in Frame C; toggles on/off via Frame C gear icon). It renders *only* escalation-class events from the current session (desktop_pause, governance_negative_regression, governance_variance_alert, static_rule_fire, hitl_timeout). Each event is a card anchored to wall-clock time. Clicking an event card opens a split-view detail modal: (Left) the **decision-diff** -- side-by-side rendering of the matched message content, the proposed action, and the confidence score at decision time. (Right) the **causal context** -- 5 prior decisions (compressed: action + agent + reason), the 3 next decisions (post-escalation), and the agent activity in the 10s window around the event. The timeline auto-scrolls to surface unacknowledged escalations; operator dismisses via swipe or X. Timeline is **non-intrusive** -- it never auto-foregrounds (per M2), it whispers via a paired label+color badge showing unread escalation *count* (e.g. "1 escalation" in amber). **Compliance hard-gates**: (1) Server-side self-exclude filter on /api/escalations endpoint (match EscalationRail.svelte createSelfExcluder pattern) plus client-side redundant defense-in-depth filter; (2) zero new bus envelope kinds introduced (consumes existing escalation_event rows from decisions table only); (3) timeline pane does NOT auto-expand on mount (operator must click badge or gear icon); (4) badge pairs text label (e.g. "1 escalation") with color (#f97316 / orange) -- color never sole signal; (5) if governance.py authorship of confidence_before/after is needed, ADR-18 amendment required first. Implementation surfaces: new GET /api/escalations?session_id&limit=100 endpoint (returns {escalation_id, event_type, triggered_at, message_id, proposed_action, confidence, agent_id}); new POST /api/escalations/{id}/dismiss endpoint for operator ack; new dashboard/ui-next/src/lib/components/EscalationTimeline.svelte (collapsible card list, split-view detail modal); new dashboard/ui-next/src/lib/components/DecisionDiff.svelte sub-component (before/after action + confidence + message content side-by-side). Message_bus.py requires additive metadata linkage (message_id + decision_id tracking in escalation rows; expose prior/next decision context via server query -- purely additive, no FROZEN surface edits). Zero soak/cassette burden: existing escalation events are already recorded; query endpoint is read-only aggregation, no new envelope kind minted.

## Operator value

Transform a reactive 'what happened' into a forensic 'why did it happen, and was it right?' Audit trail becomes immediately actionable without context-switching. Session health post-mortems become possible in-UI ("we had 3 BLOCKs around minute 8 because the agent escalated destructive commands"). Forensic visibility without the cognitive load of drilling the entire decision feed.
