### Verdict — Hydrator is NOT the p95 driver

The probe falsifies the v1.1-task-plan §I "suspect" hypothesis:

- **`hydrator_inline_s`** is 0.000 ms across all 90 calls (cold + warm).
  The cross-session `Hydrator` thread runs to completion in well under a
  millisecond on the empty/near-empty patterns table, so its execution
  window never overlaps with any subsequent `evaluate()` call.
- **`engine_construct_s`** is 2-3 ms once per session — negligible at any
  level of the latency budget.
- **`registry.get_or_create`** caches after the first call; per-call
  registry hit is `~0.02 ms`.
- **`cli_subprocess_s`** dominates: p95 17–21s, mean 7–8s. This matches
  ADR-5's documented CLI cold-start shape and accounts for ~100% of the
  per-call wall time. Some calls show `cli_subprocess_s` larger than
  `wall_total_s` because the soak driver/probe attribution sums any
  `governance_call` events landing in the time window — the next-evaluate
  attribution can pull in the previous call's terminal `exited` event when
  it lands a few ms after the call returns.

The 19.08 s p95 from `reports/soak-20260502T201806Z.md` is consistent with
the per-call CLI subprocess distribution observed here. There is no
Hydrator-shaped contribution to remove.

### Action taken

Even though the profile clears the Hydrator, Task I Step 2's "Plan A"
lazy-init refactor is still applied as a defensive cleanup:

- `EngineRegistry.get_or_create` no longer spawns the Hydrator; the
  daemon thread is started by a one-shot lazy wrapper around
  `engine.evaluate` instead.
- This frees engine construction from any sync DB read (the Hydrator's
  `bus.get_cross_session_patterns()` call), guarding against future
  patterns-table growth that could turn the spawn into a real cost.
- Tests in `tests/test_hydrator_lazy_init.py` pin the contract:
  `get_or_create` does not block on Hydrator completion, and
  `engine.hydrated` transitions False -> True on a bounded timeline
  after the first `evaluate()`.

### Real p95 fix lives in Task J

The CLI subprocess cold-start IS the v1.0/v1.1 p95 driver, exactly as
ADR-5 §"v1.1 follow-up" already calls out. The substantive fix —
warm-pool of `claude` subprocesses — is **Task J** in
`docs/v1.1-task-plan.md`. Task I's lazy-init does not by itself bring
p95 below the 15s ADR-5 ceiling, and the post-fix soak verification in
this PR is expected to land at the same shape as the v1.0 final soak.
ADR-5 should not be ratcheted on the basis of this task alone.
