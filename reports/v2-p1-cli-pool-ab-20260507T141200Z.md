# v2.0 P1 — cli_pool worker A/B (per-turn isolation lever revival)

- **Date**: 2026-05-07
- **Driver**: `tools/soak_driver.py --cli-pool-size 2 --worker-recycle-every-n {unset|1|5|10}`
- **Total wall-clock**: ~130 min (4 × ~32.5 min) sequential
- **Hypothesis**: cli_pool's long-lived stream-json workers, reused across ~30 turns per soak, bias Haiku toward conversational ALLOW on later destructive prompts. Recycling workers more aggressively should revive the dormant verdict-fallback lever (`cli_dispatch_fallback_ms`).
- **Companion docs**: `docs/v2.0-task-plan.md` §P1, `docs/adr/ADR-18-mvp-surface-freeze.md` Rule 2, `reports/p1a-corpus-haiku-verdicts-20260507T083813Z.md` (v1.9 fresh-process Haiku baseline)

## Verdict

**Hypothesis falsified at all four cadences.** Fallback lever (`cli_dispatch_fallback_ms` p95 + `governance_fallback_routed` envelope emission) fires **0%** at every recycle cadence tested (status quo, N=1, N=5, N=10). cli_pool warm-process reuse is **not** the cause of fallback-lever dormancy.

**Counter-finding**: aggressive recycle *worsens* ALLOW p95 by ~2 s relative to control (status-quo reuse) without trading for any fallback fire. The status-quo reuse is the latency-correct default; no recycle cadence is recommended.

**ADR-18 Rule 2 disposition**: A/B revival probe falsified. Rule 2 §"What counts as a strike" grants **anticipatory rip authority** for the fallback path in the same cycle. v2.0 P3 will rip both the Haiku fastpath router (DORMANT-3 entering v2.0) and the verdict-fallback retry path (DORMANT-2 + falsified revival hypothesis = anticipatory rip authority).

### Latency observations

- **Arm A is the latency winner across the board.** ALLOW p95 4.17 s vs B 6.84 s / C 6.10 s / D 6.24 s — control is 2.0–2.7 s faster than every recycle cadence. Spawn overhead at ≈ 6 calls/min for fresh workers swamps any conversational-context-rotation benefit.
- **Arm A also beats v1.9 ship-gate ALLOW p95** (4.17 s vs 8.54 s, −51%). Plausible read: v1.9 ship-gate caught the upstream variance peak that ADR-5 v1.9 §Caveats already classified as upstream API round-trip drift. The four arms today were run back-to-back over ~130 min, so within-window variance dominates over cross-cycle variance. Arm A is **not** a structural improvement claim — it is the same pool code v1.9 shipped.
- **Arms B/C/D show timeout-degrade events** in the per-call subprocess fallback path (`cli_pool path failed (response timeout); degrading` in stderr; `RESPONSE_TIMEOUT_S = 25.0` in `cli_pool.py`). Fresh-worker cold-start exceeds 25 s on this hardware on every recycle. The per-call degrade in `cli_governance.py:442` falls back to `subprocess.run`, the call still completes, the pool stays alive, and the report still PASSes. **This is operationally observable but does not fire the verdict-fallback lever** — which is the lever P3 will rip — because the verdict-fallback lever is gated on Haiku verdict / confidence, not on cli_pool transport health.

### Decision-action distribution

All 4 arms × 60 events = 240/240 ALLOW. Zero BLOCK / INTERVENE / SUGGEST verdicts surfaced from cli_governance under any cadence. The verdict-fallback lever (verdict==ENGAGE branch, v1.9 P1) cannot fire if no event terminates at a non-ALLOW verdict. The corpus + warm-pool + fresh-pool combinations exhausted in this A/B cannot trip the lever.

### Falsification scope

What this A/B falsifies:
- **cli_pool warm-process reuse is not the cause of fallback-lever dormancy.** The lever fires 0% at every cadence between "no recycle" and "every-turn recycle". If reuse were the cause, N=1 would have revived it.

What this A/B does **not** falsify:
- **Corpus-weakness hypothesis** is already separately falsified by `reports/p1a-corpus-haiku-verdicts-20260507T083813Z.md` (fresh-process Haiku BLOCKs 100% of wrapped destructive prompts at confidence ≥ 0.85). The remaining gap is between "fresh-process probe BLOCKs at 100%" and "soak driver runs 60/60 ALLOW". The P1a probe used `_wrap_user_prompt` to wrap each prompt with a destructive-corpus framing instruction; the soak driver does not wrap prompts the same way. The soak driver's `_L2_L3_TRIGGER` corpus produces wrapped-style framings inside the prompt body, but Haiku's first-turn ALLOW interpretation persists across the soak.
- **The wrapping/framing gap is the next investigation lever**, not part of v2.0. ADR-18 Rule 4 caps phases at 3; v2.0 P1 / P2 / P3 are at the cap. This finding lands in v2.1 backlog as a 🟡 corpus-framing parity item.
